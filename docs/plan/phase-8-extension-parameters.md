# Phase 8 ‚Äî extension-parameters

## Header

- **Status:** ready ‚Äî Plan DoR
- **Goal (one line):** Additively let a custom extension declare named, typed, optional/required **input parameters** ‚Äî via a class attribute ‚Äî that callers pass by name at the call site (`@grid(Photos, columns: 4)`), reusing the definition-prop machinery so the call shape is byte-for-byte identical to a definition with props (`@card(Article, style: "wide")`).
- **Depends on:** nothing.
- **Changes an externally-visible contract:** yes, additively only ‚Äî (a) a new **extension-class attribute** by which a custom extension declares typed named parameters (a genuinely new declaration surface no existing extension uses); (b) the existing **`HED5005`** ("named arguments not supported") error is **additively relaxed** on the new surface ‚Äî a named argument now binds instead of erroring **only** when the target extension declares a matching parameter; an extension that declares none still rejects named arguments with `HED5005` unchanged; (c) a build-time-only malformed-attribute diagnostic takes the next free generator id **`HED7017`**. No template valid today changes its output or its diagnostics.

## Goal

A definition can already declare **typed named props** alongside its positional model ‚Äî `<card(style: string = "plain", compact: bool = false)>` ‚Äî passed by name at the call site: `@card(Article, style: "wide")` ([Props](../language-reference.md#props-nameprop-type--default)). A **custom extension** ([`IExtension`](../custom-extensions.md)) has no equivalent: it receives its positional context/model (the `[DataType]` value) and the chained value, but **no first-class, typed, *named-parameter* channel**. Its `Scope` does carry more than those two ‚Äî `ParentModelData`, `CallerData`, and the `[ScopeChannel]` `TryRead` local-context channel ([custom-extensions ‚Üí The Scope](../custom-extensions.md#the-scope)) ‚Äî but none of those is a *caller-declared, typed, named input* the way a definition's props are. An author who wants a `@grid(...)` extension parameterised by a column count has no first-class, typed, named way to pass that count ‚Äî only the positional model channel, which is the wrong shape for a configuration knob.

This phase closes that asymmetry **additively**. A custom extension declares named, typed input parameters as an attribute on the extension class (illustrative shape, exact API a spec detail: `[Prop("columns", typeof(int), Default = 3)]`). Each parameter has a name and a type, and is **optional** when it carries a default or **required** when it does not ‚Äî mirroring definition-prop semantics exactly. The caller passes them by name after the positional model ‚Äî `@grid(Photos, columns: 4)` ‚Äî where `Photos` is the extension's context/model (exactly as today) and `columns: 4` is a declared named parameter. That call is **byte-for-byte the same call shape** as a definition with props; the symmetry at the call site is the whole point of the phase. The extension's C# reads its declared parameter values at render time from the `Scope` it already receives. The user-visible outcome: custom extensions gain the same typed, named, validated parameter contract definitions already have, with one uniform call-site syntax across definitions and extensions.

## Non-goals / scope boundary

- **No content regions or slots on extensions.** This phase gives extensions **parameters** (typed named *values*), not overridable content. An earlier exploration that would have given extensions overridable content regions/slots is **rejected and out entirely** ‚Äî an extension is C# code with no template body to host regions, so the region model does not apply to it. Extensions get parameters plus their existing context/model channels; nothing about template regions.
- **No change to definition props.** Typed named props on definitions ([Props](../language-reference.md#props-nameprop-type--default)) are unchanged; this phase **reuses** their machinery and extends its population source to extensions ‚Äî it does not alter prop semantics, defaults, inheritance, or diagnostics for definitions.
- **No change to Phase 7's definition content-regions.** [Phase 7](phase-7-named-content-regions.md) (named content regions on definitions) is a separate, unrelated feature; this phase neither depends on it nor touches it.
- **No named regions on built-in extensions.** Built-in verbs (`@if`, `@list`, `@date`, ‚Ä¶) are unchanged; this phase adds a *declaration surface* custom extensions may opt into, not new inputs to the built-ins.
- **No change to the positional model / chained channels.** `[DataType]` (the positional model) and `[ChainedType]` (the chained value) keep their exact meaning; parameters are an **additional**, named, opt-in channel, not a replacement for either.
- **Phases 1‚Äì7** are unrelated increments and are not touched here.

The seam this marks: parameters are declared inputs of *value* type. If a future phase ever wanted extensions to accept *content* (a body region), that would be new work with its own design ‚Äî this phase deliberately does not open it.

## Design direction

**Chosen approach: mirror the existing definition-prop *contract* onto custom extensions ‚Äî reuse the internal `PropLayout`/`PropsBinder` table and binder in the layers that already run it, and reimplement parameter resolution over Roslyn symbols in the generator exactly as it already does for definition props ‚Äî do not invent a new concept.** Definition props already solved every hard part of "typed named parameters passed by name": the [`PropLayout`](../../src/Heddle/Runtime/Expressions/PropLayout.cs) table that describes a target's declared parameters, the [`PropsBinder`](../../src/Heddle/Runtime/PropsBinder.cs) that binds call-site named arguments against that table, and the positioned diagnostics `HED5001`‚Äì`HED5004`. Both types are `internal sealed` to `Heddle`, so they are shared **directly** only by the three layers that link `Heddle` ‚Äî the **dynamic runtime**, the **LSP**, and the **precompiled runtime**. The **generator** emits code into the *user's* assembly and cannot see those internals, so it already **reimplements** the same resolution over Roslyn symbols ‚Äî its `PropLayoutInfo`, commented "Reimplements `PropLayout.Resolve` over symbols", in [`TemplateEmitter`](../../src/Heddle.Generator/Emit/TemplateEmitter.cs). The whole design of this phase is to make a custom extension a **second source** for that one conceptual contract: a new population source into the shared internal table for the three linking layers, **and** a parallel new attribute-reading path in the generator alongside the reimplementation it already carries ‚Äî so an extension's declared parameters flow through the identical binding-and-validation *contract* a definition's props already do, realized per layer exactly as props already are.

Three decisions define the model, each chosen because it reuses a proven Heddle mechanism rather than adding a parallel one:

- **Declaration is a class attribute, one per parameter.** A custom extension declares each parameter with an attribute on the extension class (illustrative: `[Prop("columns", typeof(int), Default = 3)]`). This matches the existing **one-attribute-per-concern** extension-attribute family ‚Äî `[ExtensionName]`, `[DataType]`, `[ChainedType]`, `[EncodeOutput]`, `[BranchRole]` ([custom-extensions ‚Üí Attributes](../custom-extensions.md#attributes)) ‚Äî so a parameter declaration reads like the attributes authors already write. A parameter with a default is optional; one with no default is required ‚Äî the same rule definition props use ([Props](../language-reference.md#props-nameprop-type--default)). The exact attribute API (name, type, default, required marker) is a residual spec detail; the lean is per-parameter `[Prop(name, type, default)]`.

- **The call site is the definition-with-props call, verbatim.** `@grid(Photos, columns: 4)` uses the same lexer/parser production and the same named-argument list as `@card(Article, style: "wide")`. `Photos` binds to the extension's positional model exactly as today; `columns: 4` is a native-expression value evaluated in the caller's context and bound to the declared `columns` parameter. There is **no new call-site grammar** ‚Äî the named-argument surface already exists; this phase widens *what target* is allowed to receive it. Call-site symmetry is not a nice-to-have; it is the deliverable.

- **Reading parameters is via the `Scope` the extension already has.** A definition reads a prop by bare name in its body (`@(style)`); an extension is C# and has no template body, so it reads its declared parameter values at render from the `Scope` passed to `ProcessData`/`RenderData` ‚Äî the same object from which it already reads `ModelData`, `ChainedData`, and the local-context channel (`TryRead`) ([custom-extensions ‚Üí The Scope](../custom-extensions.md#the-scope)). The precise accessor is a spec (HOW) detail; the lean is a params accessor on `Scope` alongside the existing fields, keyed by the declared parameter name.

Why this beats the alternatives: **a bespoke extension-parameter mechanism** (a second table, a second binder, a second set of diagnostics) was rejected as duplicated surface ‚Äî the prop path already does exactly this and is already threaded through every backend. **Overloading the positional model** (packing configuration into the `[DataType]` value) was rejected because it is untyped-by-convention, unnamed, and reads nothing like the definition contract it should mirror. **Reusing `[ScopeChannel]` publish/read** was rejected because that is *sibling* coordination inside one body execution, not a *caller‚Üícallee* typed input. Mirroring props is the only option that gives extensions the exact same author-facing contract definitions have, at the exact same call site, for near-zero new concept.

**Why the reach is contained: the same parameter contract is already realized in every layer.** Props already run in all four backends ‚Äî the **dynamic runtime**, the source **generator** ([`Heddle.Generator`](../../src/Heddle.Generator/Emit/TemplateEmitter.cs)), the **LSP** ([`Heddle.LanguageServices`](../../src/Heddle.LanguageServices)), and **precompiled binding** ‚Äî three of them over the internal [`PropLayout`](../../src/Heddle/Runtime/Expressions/PropLayout.cs)/[`PropsBinder`](../../src/Heddle/Runtime/PropsBinder.cs), and the generator over its **own parallel Roslyn-symbol reimplementation** (the same honest split [Phase 7](phase-7-named-content-regions.md) records, where "the generator carries its own parallel tracking"). This phase adds a **new population source** ‚Äî an extension's `[Prop]` attributes ‚Äî into that existing contract: for the three linking layers it is a new source into the shared internal table (no new pipe); for the generator it is a **genuinely new attribute-reading path** parallel to the definition-prop reimplementation it already carries. The rollout is therefore the **same wave** props already established ‚Äî all four layers together, so the dynamic and precompiled backends stay at parity from day one. It is **somewhat smaller** than [Phase 7](phase-7-named-content-regions.md) ‚Äî the call-site binding, the diagnostics, and the non-generator table all already exist, and only their *source* widens for those layers ‚Äî but it is **not** merely "widen the source": the generator needs a real new attribute-reading path over symbols, so the honest framing is Phase 7's (generator carries a parallel path), not a table-free freebie.

## Dependencies & ordering

**Nothing must land first.** This phase mirrors the *existing* definition-prop infrastructure ‚Äî [`PropLayout`](../../src/Heddle/Runtime/Expressions/PropLayout.cs) / [`PropsBinder`](../../src/Heddle/Runtime/PropsBinder.cs) / `HED5001`‚Äì`HED5006` ‚Äî which is already stable in the tree today. It is **independent of [Phase 7](phase-7-named-content-regions.md)**: Phase 7 delivers definition content-regions, a different feature with a different table; this phase touches neither regions nor `@out()`. It is an additive, self-contained feature; it is listed last in the plan as a discrete deliverable, but it carries no hard technical dependency and could land at any point once the prop machinery it mirrors is in place (which it already is).

**What this phase unblocks:** nothing downstream depends on it. It completes the "typed named parameters" story by extending it from definitions to custom extensions.

## Back-compat / impact

Governed by [R5](common/standing-rulings.md#standing-rulings-user-set) (additive by default) and the byte-identity back-compat gate. This feature takes **no** breaking-window change ‚Äî it is purely additive.

- **The `HED5005` relaxation touches no template valid today.** Today, passing named arguments to a call that is **not** a prop-declaring definition is a compile error: `HED5005` (`NamedArgumentsNotSupported`) when the target is an extension/function/unknown, `HED5006` (`DefinitionHasNoProps`) for a prop-less definition ([HeddleDiagnosticIds](../../src/Heddle/Data/HeddleDiagnosticIds.cs); [Props](../language-reference.md#props-nameprop-type--default)). Because passing a named argument to an extension is an error **today**, no template that does so compiles today ‚Äî so relaxing that error can only make a **previously-erroring** shape start compiling. It cannot change the behaviour of any template that is valid today. This is the same shape of additivity as the `@@` literal escape in [Phase 2](phase-2-authoring-ergonomics.md), which *removes* a compile error rather than changing a working template.
- **The relaxation is keyed strictly on the extension declaring a matching parameter.** `@grid(x, columns: 4)` compiles **only** when `grid` declares a `columns` parameter. An extension that declares **no** parameters still rejects named arguments with **the same `HED5005`, on the same condition** ‚Äî same id, same positioned failure. Its *message text* is updated post-phase (today it reads "‚Ä¶is not a definition ‚Äî named arguments can only be passed to definitions that declare props," which is no longer literally true once a parameter-declaring extension accepts them), so the claim is same-id-same-condition, not byte-for-byte message identity. So the relaxation is scoped to the new declaration surface; it never silently accepts a stray named argument on an unparameterised extension.
- **Definition props are unchanged.** `HED5005`/`HED5006` keep their exact meaning for definitions; the reused `HED5001`‚Äì`HED5004` keep their meaning and are simply now reachable when the target is a parameter-declaring extension.
- **Rendered output is byte-identical.** No template valid today changes its bytes; the differential corpus that asserts dynamic == precompiled does not move on any existing fixture, and the new parameter fixtures render identically under both backends.
- **Same four-layer wave as props; bodiless binds at build, bodied reuses the existing fallback.** The dynamic runtime, generator, LSP, and precompiled binding all learn the new population source together, so parity holds from day one. A **bodiless** parameter-declaring call (`@grid(Photos, columns: 4)`) binds its named arguments at build time on the precompiled tier, exactly as a bodiless definition-with-props call does ‚Äî full parity. A **bodied** parameter-declaring call (`@grid(x, columns: 4){{‚Ä¶}}`) follows the **existing** bodied-custom-call contract: it falls back to the dynamic tier ([custom-extensions ‚Üí Precompiled mode](../custom-extensions.md#precompiled-mode)), where byte-parity is preserved *by* that fallback (the dynamic tier renders it). There is therefore no *new* degrade-to-dynamic path ‚Äî bodiless is build-time parity, bodied reuses the fallback custom calls already have.
- **Docs:** [custom-extensions](../custom-extensions.md) and the [language-reference Props](../language-reference.md#props-nameprop-type--default) section gain the extension-parameter note. (Doc edits are downstream of the code change, not part of this WHAT.)

## Success criteria

Measurable, checkable statements a spec could turn into a test. Back-compat first.

- [ ] **No template valid today changes output.** Every existing fixture in the differential corpus renders byte-identically under **both** the dynamic and precompiled backends after this phase; the dynamic == precompiled differential does not move.
- [ ] **Parameter-less extension still rejects named args.** `@ext(ctx, p: v)` on an extension that declares **no** parameters is still `HED5005` (`NamedArgumentsNotSupported`), positioned, on the same condition as today (its message wording is updated, but the id and the failure are unchanged) ‚Äî proving the relaxation is scoped to the new surface.
- [ ] **Parameter-declaring extension compiles and reads the parameter.** A custom extension that declares a `columns` parameter, called `@grid(Photos, columns: 4)`, compiles, and the extension's C# reads `columns` = `4` at render (the rendered output reflects the value).
- [ ] **Optional vs required mirror prop semantics.** A declared parameter **with** a default is optional (an all-defaults call `@grid(Photos)` compiles and renders with the default); a declared parameter **without** a default is required (omitting it is `HED5002`, `MissingRequiredProp`, positioned) ‚Äî matching definition-prop behaviour.
- [ ] **Unknown / mistyped / duplicate parameter reuse the existing diagnostics.** On a parameter-declaring extension: an undeclared name is `HED5001`, a value whose type does not convert is `HED5003`, and the same name twice is `HED5004` ‚Äî the existing prop diagnostics, positioned, now reachable for extensions.
- [ ] **Call site is byte-for-byte identical to a definition-with-props call.** A fixture that swaps a **prop-declaring definition** for a **parameter-declaring extension** with the *same* parameter signature and *identical* call bodies (e.g. `@grid(Photos, columns: 4)`) parses, binds, and diagnoses identically ‚Äî the same call text is accepted, and the same malformed variants draw the same `HED5001`/`HED5003`/`HED5004`.
- [ ] **Dynamic == precompiled parity on the new fixtures.** Every new parameter fixture renders byte-identically under the dynamic and precompiled backends.
- [ ] **Malformed `[Prop]` attribute is diagnosed on both tiers, never silently.** A malformed parameter declaration on an extension class draws a positioned diagnostic on **both** tiers: on the **dynamic tier** the reused declaration-side prop id (`HED5007` duplicate declaration / `HED5009` default not convertible / `HED5010` unresolved type / `HED5015` reserved name, as applicable), and on the **generator/build tier** its twin `HED7017` ‚Äî never a silent failure, never a runtime crash. (Criterion 8 is testable on both tiers.)

## Validation scenarios

The scenarios use a `grid` extension declaring one parameter `columns: int = 3` (illustrative attribute: `[Prop("columns", typeof(int), Default = 3)]`), paired for the symmetry check with a definition `<grid(columns: int = 3)>`.

| Input | Expected outcome |
|---|---|
| `@grid(Photos, columns: 4)` where `grid` is an extension declaring `columns: int = 3`. | Compiles; the extension reads `columns` = `4` at render and lays out four columns. `Photos` binds to the positional model exactly as today. |
| `@grid(Photos)` (all defaults) on the same extension. | Compiles; the extension reads `columns` = `3` (the declared default). Confirms optional-with-default semantics. |
| `@ext(ctx, p: v)` where `ext` is an extension that declares **no** parameters. | Positioned compile error `HED5005` (`NamedArgumentsNotSupported`) ‚Äî unchanged from today. |
| `@grid(Photos, rows: 4)` ‚Äî a name the extension does **not** declare. | Positioned compile error `HED5001` (unknown parameter; message lists the declared parameters), now for an extension. |
| `@grid(Photos, columns: "x")` ‚Äî value type does not convert to `int`. | Positioned compile error `HED5003` (type mismatch), now for an extension. |
| `@grid(Photos, columns: 4, columns: 5)` ‚Äî same name twice. | Positioned compile error `HED5004` (duplicate argument), now for an extension. |
| A `grid` extension declaring a **required** parameter (`span: int`, no default), called `@grid(Photos)` with `span` omitted. | Positioned compile error `HED5002` (`MissingRequiredProp`), matching definition-prop behaviour. |
| **Symmetry fixture:** the same call bodies against a **definition** `<grid(columns: int = 3)>` and an **extension** `grid` declaring the identical signature. | The `@grid(Photos, columns: 4)` call text and every malformed variant (`rows:`, `columns: "x"`, duplicate) are accepted / diagnosed identically across the two targets ‚Äî proving byte-for-byte call-site symmetry. |
| A malformed `[Prop]` declaration on an extension class (unresolvable type / non-convertible default / duplicate or reserved parameter name). | Positioned diagnostic, no silent failure: on the **dynamic tier** the reused declaration-side prop id (`HED5010` unresolved type / `HED5009` default not convertible / `HED5007` duplicate declaration / `HED5015` reserved name, as applicable), and its **build-time twin** `HED7017` on the generator tier. |
| Any existing template (parameter-less extensions and prop-declaring definitions), rendered under both backends. | Byte-identical to today, dynamic and precompiled; differential unchanged. |

## Open questions

None ‚Äî resolved (see [open-questions.md](open-questions.md)). Every P8 question (P8-Q1‚ÄìP8-Q4) is folded into the goal, design direction, back-compat, and success criteria above; each is a residual spec-level (HOW) item on a settled WHAT, carried with a lean, so the phase is at Plan DoR. The register identifiers map to the earlier draft labels as Qa‚Üí**P8-Q1**, Qb‚Üí**P8-Q2**, Qc‚Üí**P8-Q3**, Qd‚Üí**P8-Q4**:

- **P8-Q1 ‚Äî the attribute API shape for declaring an extension parameter** ‚Äî resolved lean: a per-parameter class attribute `[Prop(name, type, default)]` (illustrative), matching the one-attribute-per-concern extension-attribute family (`[ExtensionName]`/`[DataType]`/`[ChainedType]`/`[EncodeOutput]`/`[BranchRole]`). Residual (HOW): the exact declaration syntax the generator and LSP read.
- **P8-Q2 ‚Äî how the extension's C# reads declared parameter values at render** ‚Äî resolved lean: a params accessor on the `Scope` the extension already receives (alongside `ModelData`/`ChainedData`/`TryRead`), keyed by parameter name. Residual (HOW): the render-time read API surfaced to extension authors.
- **P8-Q3 ‚Äî required vs optional parameters for extensions** ‚Äî **decided (mirror definition props):** a parameter with no default is required (unbound ‚Üí `HED5002`), one with a default is optional ‚Äî the same rule definition props use, which is the whole thesis of this phase. Not open.
- **P8-Q4 ‚Äî the malformed-`[Prop]` diagnostic** ‚Äî resolved lean: a malformed declaration is diagnosed on **both** tiers ‚Äî the **dynamic tier** reuses the declaration-side prop ids `HED5007`/`HED5009`/`HED5010`/`HED5015` (now reachable when the declaration source is an extension attribute), and the **generator/build-time twin** takes the next free generator id `HED7017` (highest today is `HED7016`). Residual (HOW): the exact condition-to-id mapping and the registry claim.
