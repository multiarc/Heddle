# Phase 5 — docs-and-benchmark-credibility

## Header

- **Status:** ready — Plan DoR
- **Goal (one line):** The repository states one coherent version status everywhere, has no dangling spec links, and backs its performance claim with more than one workload.
- **Depends on:** nothing — editorial and additive; can land independently of the other phases.
- **Changes an externally-visible contract:** no — documentation and the internal benchmark harness only; no engine, API, generated-output, or rendered-byte behavior changes (standing ruling R5).

## Goal

Two credibility gaps in the current tree undercut work whose engineering is otherwise sound, and this phase closes both without touching engine behavior.

First, the repository contradicts itself about whether **2.0** has been released. Several docs already assert or imply a released 2.0.0 — [docs/README.md](../README.md) ("Current version: **2.0.0**"), [building.md](../building.md) ("The current release line is **2.0.0**"), the two migration guides ("verified to compile against **Heddle 2.0.0**"), [built-in-extensions.md](../built-in-extensions.md) and [csharp-api.md](../csharp-api.md) ("byte-identical to 2.0.0"), and the spec records/index ("the released 2.0.0 source", "Shipped in 2.0.0") — while [CHANGELOG.md](../../CHANGELOG.md) still carries the 2.0 contents under `[Unreleased]` and states they have "not shipped yet", and `git tag -l` shows only `v1.0.0`/`v1.0.1`. The assessment flags the inconsistency in [§2](../language-assessment.md). The user has confirmed **2.0 is releasing imminently** (P5-Q1), so this phase reconciles the docs *toward* `2.0.0` as the current release line — making the many docs that already say 2.0.0 correct-and-consistent and wording everything so it is ready for the release the moment the adjacent release action lands (the `v2.0.0` tag cut, the package publish, and the `CHANGELOG` `[Unreleased]` → dated `[2.0.0]` move). Separately, the spec docs carry dead links: [docs/spec/README.md](../spec/README.md) calls two initiatives "ready for implementation" while linking four files that were deleted in the `Clean-up` commit, and the living [cross-cutting-decisions.md](../spec/common/cross-cutting-decisions.md) registry still links a deleted `import-removal-spec.md`. The outcome of this phase is a single, 2.0.0-forward source-of-truth version statement applied consistently, and spec docs whose links all resolve — so a reader who greps the docs finds no self-contradiction and clicks no dead link.

Second, the performance story rests on exactly one workload: a ~55.5 KB composition-heavy page (`home.heddle` + `layout.heddle`) under a byte-identical parity assertion, documented in the [README Performance section](../../README.md#performance) and the committed [2026-07-11 run](../benchmarks/2026-07-11). The assessment ([§5 observation 2](../language-assessment.md), [§8 item 5](../language-assessment.md)) is candid that one workload is not a characterization and invites the obvious rebuttal — "you only benchmarked your best case." The outcome is a broadened evidence base: at least a trivial-substitution workload (the shape where the assessment predicts Heddle's advantage narrows or inverts) and a large-loop workload, published alongside the existing run with the same parity discipline and the same honest, workload-dependent framing.

## Non-goals / scope boundary

- **Cutting the `v2.0.0` release or tag.** That is a release action, not an editorial one, and stays out of this phase's scope. The maintainer has confirmed 2.0 is releasing imminently (P5-Q1), so this phase words the docs to be *ready and internally consistent for* `2.0.0` and flags the tag-cut + package publish + `CHANGELOG` `[Unreleased]`→`[2.0.0]` finalization as the separate, adjacent action that must accompany them — it does not perform that action here, so the docs are not published asserting a tag that does not yet exist.
- **Any engine, generator, or runtime change.** No source that affects compilation or rendered bytes is touched; the only code in scope is the benchmark harness under `src/Heddle.Performance` (test fixtures and benchmark twins).
- **Rewriting the benchmark harness or the parity mechanism.** New workloads reuse the existing precedented shape; they do not restructure `ParityCheck`, the runner twins, or the reporting.
- **Competing on micro-benchmarks or adding competitor engines.** The goal is characterization and honesty, not a marketing win; the existing four-engine (+ indicative Razor) set is unchanged, and a workload predicted to be unfavorable to Heddle is published rather than suppressed.
- **Editing the append-only historical spec records.** [records.md](../spec/records.md) is governed by its own "entries are never edited or deleted" discipline; reconciling its wording/links is bounded by that rule (resolved as P5-Q4) and by default is left to the living index rather than performed in the historical rows. This is distinct from [cross-cutting-decisions.md](../spec/common/cross-cutting-decisions.md), which is a **living** registry (not append-only for links) and so **is** in scope for the dead-link fix.
- **Seam to later work.** Closing the contextual-encoding gap (assessment §8 item 2) and any actual 2.0 release are separate efforts; this phase only makes the documentation and evidence match reality.

## Design direction

**Version-status reconciliation — align toward the imminent 2.0.0 release (user decision, P5-Q1).** The maintainer has confirmed 2.0 is **releasing imminently**, so the canonical statement reconciles the docs *toward* a released `2.0.0` rather than toward "unreleased". Define one canonical statement and apply it everywhere a version status is asserted or implied, so the docs already assert `2.0.0` consistently the moment the release action lands. The two facts the docs must stop conflating are still separated — but now in the release-forward direction: the docs describe **2.0.0 as the current release line**, and the **release action** that makes that literally true (cutting the `v2.0.0` git tag, publishing the package, and moving the `CHANGELOG` `[Unreleased]` section to a dated `[2.0.0]` entry) is the trigger this plan's wording anticipates rather than performs. The many docs that already say "2.0.0" (README, building, migration guides, extension/API references) become correct-and-consistent; the `CHANGELOG`'s "not shipped yet" line is reconciled as part of the release cut. This is the ratified direction; it supersedes the earlier lean toward "unreleased" (which was correct only while no release was intended). Because the tag/package/`CHANGELOG`-finalization is a release act, this plan's editorial scope makes the docs *ready and internally consistent for* 2.0.0 and flags the release cut as the separate, adjacent action that must accompany them (so the docs are not published asserting a tag that does not yet exist).

**Dangling links — follow the precedent already in the index, across every living doc that carries them.** [docs/spec/README.md](../spec/README.md) already retired a comparable initiative cleanly: the "v2.0 nine-phase effort" row reads "Shipped in 2.0.0; specs retired … removed from the tree (git history only)." The two initiatives whose plan/spec files were deleted (the post-2.0 improvement effort and the legacy `@import` removal — both of whose shipped residue is real, per the CHANGELOG and the records ledger) are repointed to that same "specs retired (git history)" treatment and their dead links removed, so the index is internally consistent with how it already describes retired specs. The **same fix extends to the living [cross-cutting-decisions.md](../spec/common/cross-cutting-decisions.md) registry**, whose diagnostic-ownership row still links the deleted `import-removal-spec.md` (`HED4003` at line 159) — that registry is not append-only for links, so the dead link is repointed (to the amendments ledger / git history) rather than left dangling. The append-only historical `records.md` rows are the sole exception (see the non-goal above).

**Benchmark breadth — extend the precedented harness, don't rebuild it.** The harness already supports both patterns a new workload needs: parity-checked (a `.heddle` fixture under [TestTemplates/](../../src/Heddle.Performance), matching competitor twins, a `Runners/*Test.cs` twin registered in [`ParityCheck.Twins()`](../../src/Heddle.Performance/Runners/ParityCheck.cs), and a `[Benchmark]` in a `[MemoryDiagnoser]` class like [TextRenderBenchmarks](../../src/Heddle.Performance/TextRenderBenchmarks.cs)) and Heddle-only (a class like `BranchRenderBenchmarks`). Two new workloads bracket the existing composition page: a **trivial-substitution** workload (scalar member output dominating, minimal composition — the shape the assessment predicts narrows or inverts the gap, and the shape vendor suites such as Scriban's own favor) and a **large-loop** workload (output dominated by one large iteration). Results are published alongside the [2026-07-11 run](../benchmarks/2026-07-11) with the same committed raw BenchmarkDotNet artifacts, the same "hardware- and date-specific; reproduce yourself" framing, and — for parity-checked workloads — the same byte-identical assertion that runs in `[GlobalSetup]` before any timing. Publishing the *unfavorable* case is the credibility move, and it is cheap precisely because the harness already accommodates it.

## Dependencies & ordering

Nothing must land first. Both workstreams are additive and touch disjoint areas (documentation prose; benchmark fixtures/twins), so this phase can land in any order relative to the others and is naturally a late credibility-and-cleanup pass. It unblocks nothing structurally, but it removes two of the specific "what holds it back" items the assessment calls out (internal-docs disagreement in [§8 item 3](../language-assessment.md); single-workload evidence in [§8 item 5](../language-assessment.md)), which is the point.

## Back-compat / impact

- **Documentation.** Externally visible prose changes: readers see one version status instead of several, and the spec index links all resolve. No installable artifact, package version, or API surface changes. The `docs/README.md` version line and the migration-guide compile-target notes change wording but not the behavior they describe.
- **Benchmark harness.** New fixtures, twins, and benchmark classes are added under `src/Heddle.Performance`; existing benchmarks, the parity contract, and the 2026-07-11 artifacts are untouched. The suite is a developer-facing tool, not a shipped contract, so nothing downstream breaks.
- **No rendered-byte or engine-behavior change** anywhere (R5).

## Risks & mitigations

| Risk | Mitigation | Size (S/M/L) |
|---|---|---|
| The docs are reconciled to assert `2.0.0` but the adjacent release action (tag cut, publish, `CHANGELOG` finalization) lags, so for a window the docs assert a release line whose `v2.0.0` tag does not yet exist. | The reconciliation is scoped to be *ready for* the imminent release (P5-Q1); the canonical statement is a single, well-scoped wording, and the phase explicitly names the tag-cut / publish / `CHANGELOG`-finalization as the required adjacent action so it is not forgotten. If release slips materially, that one wording is the single place to adjust. | S |
| Fixing wording/links in the append-only [records.md](../spec/records.md) would violate its "never edited or deleted" discipline. | Scope the fix to the living index ([spec/README.md](../spec/README.md)) and user-facing docs; for records.md prefer an appended dated clarification over editing historical rows, and treat its "2.0.0" as spec-vocabulary for the branch feature set (records.md already disclaims the tag). Recorded as Q4. | M |
| The trivial-substitution workload shows Heddle at parity or slower, read as a regression. | This is the *predicted* outcome ([assessment §5 obs 2](../language-assessment.md)); the framing states workload-dependence up front, so publishing it is the credibility win, not a loss. | S |
| Authoring parity twins for two more workloads across four engines is effort and a new drift surface. | Parity is enforced in `[GlobalSetup]` and fails loudly on drift; the twin patterns are reused, not invented. If cost is prohibitive, Heddle-only is a precedented fallback (Q2). | M |
| New numbers are hardware/date-specific and could be read as absolute. | Retain the existing "hardware- and date-specific; reproduce yourself" framing and commit raw artifacts, exactly as the 2026-07-11 run does. | S |

## Success criteria

Measurable, checkable statements a spec could turn into a test.

- [ ] A single canonical version-status statement — asserting `2.0.0` as the current release line — is written down once, and every doc that states or implies a version status matches it: specifically the loci reconciled at [docs/README.md](../README.md) (line 19), [building.md](../building.md) (line 38), [coming-from-liquid.md](../coming-from-liquid.md) (line 8), [coming-from-razor.md](../coming-from-razor.md) (line 8), [built-in-extensions.md](../built-in-extensions.md) (line 660), and [csharp-api.md](../csharp-api.md) (lines 193–195).
- [ ] A grep across `docs/` for version claims (`2.0.0`, "current version", "release line") finds every hit asserting `2.0.0` as the current release line consistently, and **no** doc contradicting the canonical 2.0.0 statement.
- [ ] The `CHANGELOG` finalization to a dated `[2.0.0]` entry is identified as the release action accompanying this reconciliation; after the docs are reconciled, no doc is left contradicting the canonical 2.0.0 statement (the tag cut / publish / `CHANGELOG` move is the adjacent action this phase anticipates, not performs).
- [ ] The spec docs contain no markdown link to a nonexistent file: in [docs/spec/README.md](../spec/README.md) the four retired targets (`improvement-plan.md`, `improvement-spec.md`, `import-removal-plan.md`, `import-removal-spec.md`) are repointed or removed, and in the living [cross-cutting-decisions.md](../spec/common/cross-cutting-decisions.md) registry the deleted `import-removal-spec.md` link (line 159) is repointed; a link-check of both files reports zero dangling targets.
- [ ] The benchmark suite includes a **trivial-substitution** workload and a **large-loop** workload, each with committed BenchmarkDotNet reports (md/csv/html) published alongside the [2026-07-11 run](../benchmarks/2026-07-11), and — where parity-checked — a twin registered in [`ParityCheck.Twins()`](../../src/Heddle.Performance/Runners/ParityCheck.cs) whose assertion passes in `[GlobalSetup]`.
- [ ] The published results for all three workloads retain honest, workload-dependent framing: each workload's shape is described, no claim of universal superiority is made, and the trivial-substitution result is presented as the case where the gap narrows or inverts (per [assessment §5 obs 2](../language-assessment.md)).
- [ ] The final diff touches only documentation and `src/Heddle.Performance`; no engine/generator source, no rendered-byte behavior, and no release tag change.
