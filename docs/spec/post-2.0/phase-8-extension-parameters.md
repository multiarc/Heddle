## Scope and goal

Additively let a **custom extension** ([`IExtension`](../../../src/Heddle/Runtime/IExtension.cs)) declare named, typed input **parameters** — via a class attribute — that the caller passes by name at the call site (`@grid(Photos, columns: 4)`), reusing the definition-prop contract so the call shape is byte-for-byte identical to a definition with props (`@card(Article, style: "wide")`). Each parameter is **optional** when it carries a default and **required** when it does not — mirroring definition-prop semantics exactly. The extension's C# reads its declared parameter values at render time from the `Scope` it already receives.

**In scope:** a new `[Prop]` extension-class attribute (the declaration surface); reading it in all four layers props already reach — the **dynamic runtime** (via [`PropLayout`](../../../src/Heddle/Runtime/Expressions/PropLayout.cs)/[`PropsBinder`](../../../src/Heddle/Runtime/PropsBinder.cs)), the **LSP** (transitively through `HeddleCompiler.Compile`), the **generator** (a parallel new attribute-reading path over Roslyn symbols alongside its existing `PropLayoutInfo`), and **precompiled binding** (a build-time bind for bodiless calls, the existing fallback-to-dynamic for bodied ones); a render-time `Scope` accessor keyed by parameter name; the additive relaxation of `HED5005` and reuse of `HED5001`–`HED5004` for extensions; the malformed-declaration diagnostics on both tiers; and every fixture/golden/parity gate. **Also in scope by maintainer ruling (2026-07-15, [D8](#d8--plain-custom-extension-render-type-derives-from-encodeoutputnotencode-resolves-p8-j-e1)):** correcting the generator's **pre-existing** non-parameterized custom-extension allocation (`AllocateCustomExtension`) to derive its output render type from the extension's `[EncodeOutput]`/`[NotEncode]` — the same mapping the dynamic tier applies — instead of hard-coding `RenderType.Raw`, closing the latent cross-tier (XSS-class) divergence **P8-J-E1** for a no-parameter `[EncodeOutput]` custom extension.

**Out of scope** (plan *Non-goals*): content regions or slots on extensions (an extension has no template body to host them — rejected and out entirely); any change to definition props, to [Phase 7](../../plan/phase-7-named-content-regions.md) content-regions, to the positional model (`[DataType]`) / chained (`[ChainedType]`) channels, or to built-in extensions. **No new call-site grammar** — the named-argument production already exists; this phase widens *what target* may receive it. This spec adds **no new `HED5xxx` diagnostic id** (it relaxes `HED5005` and reuses the prop family) and exactly one new build-time id, **`HED7017`**.

## Design decisions

Every P8 plan question, lean, and pin appears here as a **closed** decision. No open questions remain.

### D1 — The `[Prop]` extension-parameter attribute (closes P8-Q1)

- **Decision.** Add `public sealed class PropAttribute : Attribute` to [`Heddle.Attributes`](../../../src/Heddle/Attributes) (new file `PropAttribute.cs`), matching the one-attribute-per-concern family:

  ```csharp
  [AttributeUsage(AttributeTargets.Class, AllowMultiple = true, Inherited = true)]
  public sealed class PropAttribute : Attribute
  {
      public PropAttribute(string name, Type type) { Name = name; Type = type; }
      public string Name { get; }
      public Type Type { get; }
      /// <summary>The default value bound when the caller omits this parameter. A non-null value makes the
      /// parameter optional. Limited to the CLR values an attribute argument may carry (primitives, string,
      /// enum, or typeof) — the same expressive range a definition prop's literal default allows.</summary>
      public object Default { get; set; }
      /// <summary>Marks the parameter optional with a null default — the only way to express an optional
      /// reference/nullable parameter whose default is null (a bare <c>Default = null</c> is indistinguishable
      /// from an unset default). Ignored when <see cref="Default"/> is non-null (already optional).</summary>
      public bool Optional { get; set; }
  }
  ```

  Usage: `[Prop("columns", typeof(int), Default = 3)]` (optional, default 3); `[Prop("span", typeof(int))]` (required); `[Prop("label", typeof(string), Optional = true)]` (optional, null default). The attribute is `AllowMultiple = true` (one per parameter) and `Inherited = true` (a subclass extension inherits its base's parameters, mirroring definition-prop inheritance and the `Inherited = true` convention of `[BranchRole]`/`[ScopeChannel]`).
- **Reading it, per layer.**
  - **Dynamic runtime & LSP:** reflection over the extension `Type` and its base-type chain — new `PropLayout.ResolveFromExtension(Type, CompileScope, string ownerDisplay, BlockPosition ownerCallPosition)` (see [WI2](#wi2)). The LSP inherits this for free because [`DocumentAnalyzer`](../../../src/Heddle.LanguageServices/DocumentAnalyzer.cs) drives `HeddleCompiler.Compile` directly (verified: `HeddleCompiler.Compile(text, compileScope, parseContext, null)` ≈45).
  - **Generator:** Roslyn symbols — new `TemplateEmitter.ResolveExtensionPropLayout(INamedTypeSymbol)` reading `[Prop]` `AttributeData` over the base-type chain (Roslyn `GetAttributes()` does not surface inherited attributes; the base-chain walk mirrors `ExtensionBinder.ReadBranchRole`). The name is `ConstructorArguments[0]` (string); the type is `ConstructorArguments[1]` (`TypedConstant` Kind `Type`, `.Value` an `ITypeSymbol`); `Default`/`Optional` from `NamedArguments`.
- **Rationale.** `[Prop("name", type)]` reads exactly like the attributes authors already write and names the concept the mechanism reuses ("prop"). A `System.Type` argument is compiler-checked and IDE-navigable — strictly better than a stringly-typed name (which the definition header needs only because it lives in template text, not C#). `Default`/`Optional` reconstruct the definition's `(HasDefault, DefaultValue)` pair unambiguously (D3).
- **`Default` conversion range (verified, no restriction needed).** An attribute argument can carry an **enum** value or a `typeof` (`Type`) — CLR forms a definition's *template-literal* default never produces — so "the same expressive range" is a superset, not an equality. The reused conversion nonetheless handles them without special-casing: `ApplyDefault` → `PropConversion.TryConvertLiteral` takes `source = Default.GetType()` and runs `CanConvertTypes(source, paramType, allowBoxToObject: true)` (`PropConversion.cs` ≈60–80/24–53). A boxed enum whose runtime type **is** the parameter type passes the identity branch (`s == t`); a `typeof` value (`RuntimeType`) passes the reference-assignability branch (`Type.IsAssignableFrom(RuntimeType)`); a mismatched enum/`typeof` default (e.g. an enum default on an `int` parameter) correctly fails → `HED5009`. `HED5009`'s `{literalType}` renders `Default.GetType().Name` (`"Mode"`, `"RuntimeType"`). No `Default`-range restriction is therefore required — the claim's "primitives, string, enum, or typeof" set is exactly what the reused path admits.
- **Alternatives rejected.**
  - *Type as a name string (`[Prop("columns", "int")]`)* — reproduces the definition header's stringly-typed form with no benefit in C#; loses compile-checking and IDE navigation; makes `HED5010` (unresolved type) common instead of near-impossible. Rejected.
  - *A single `[Props(...)]` list attribute* — breaks the one-attribute-per-concern family and reads unlike `[DataType]`/`[ChainedType]`. Rejected.
  - *A distinct name (`[Parameter]`/`[ExtensionParam]`)* — the concept **is** a prop (same table, same binder, same diagnostics); a different name would imply a different mechanism. `[Prop]` is free (Assumed state). Rejected.
- **Grounding.** [custom-extensions → Attributes](../../custom-extensions.md#attributes); [`BranchRoleAttribute`](../../../src/Heddle/Attributes/BranchRoleAttribute.cs) (`Inherited = true` precedent); plan P8-Q1.

### D2 — Render-time parameter accessor on `Scope` (closes P8-Q2)

- **Decision.** Add two public accessors to [`Scope`](../../../src/Heddle/Data/Scope.cs), keyed by parameter name, alongside `ModelData`/`ChainedData`/`TryRead`:

  ```csharp
  /// <summary>Reads a declared extension parameter by name. Returns false (never throws) when the current
  /// scope carries no parameter frame or the name was not declared.</summary>
  public bool TryGetParameter(string name, out object value);

  /// <summary>Reads a declared extension parameter by name, returning its bound value (which may be null for
  /// a null default). Throws <see cref="ArgumentException"/> when <paramref name="name"/> is not a declared
  /// parameter of the extension whose frame is active — an author bug (the name never matched a [Prop]).
  /// The no-frame case (<c>ExtensionParameters == null</c>, e.g. read outside a parameter-declaring
  /// extension's render) is one instance of "not declared" and throws identically.</summary>
  public object GetParameter(string name);
  ```

  Values are boxed `object` (the author casts, exactly as with `ModelData`) — e.g. `int columns = (int) scope.GetParameter("columns");`.
- **Carriage.** Two new **internal** `Scope` fields, threaded through every transform exactly as `PropsData`/`SlotCarrier` are, and installed only by a new internal transform:
  - `internal readonly object[] ExtensionParameterValues;` — the per-invocation bound values (the `PropsBinder.Bind`/precompiled-carriage output; the shared frozen array when all-constant, one clone otherwise).
  - `internal readonly ExtensionParameterMap ExtensionParameters;` — an immutable, per-call-site **name→index** map (a new `internal sealed class` in `Heddle.Data`; ordinal `Dictionary<string,int>` built once at bind, shared across renders and threads).
  - `internal readonly Scope WithExtensionParameters(object[] values, ExtensionParameterMap map)` — the single install point (called only by the carrier, [WI3](#wi3)).

  `TryGetParameter` resolves `ExtensionParameters.TryGetIndex(name, out i)` and returns `ExtensionParameterValues[i]`. **These fields are separate from `PropsData`** — a parameter-declaring extension called *inside* a definition body must not clobber that definition's `PropsData`, because the extension's own caller content (`@grid(…){{ @(style) }}`) is rendered through `GetInnerResult(scope)` and must still read the enclosing definition's props by index (a shared field would corrupt that read).
- **Thread-safety.** `Scope` is a `readonly struct` passed by `in`; the map is immutable and shared; the values array is never written after `Bind` returns. No per-render state lives on the extension instance (the carrier holds only the immutable binder/prototype/map). Concurrent renders of one call site share the map and, for all-constant sites, the frozen values array — read-only for both.
- **Rationale.** An extension is opaque C# with no template body compiled against indices, so it must read **by name**; a name-addressable carrier is intrinsic to that difference, not a duplicated mechanism (the *binding/validation* contract — `PropLayout`/`PropsBinder`/`HED5001`–`HED5004` — is fully reused; only the render addressing differs, as it must). Separate fields preserve `PropsData` for the extension's caller content.
- **Alternatives rejected.**
  - *Reuse `Scope.PropsData` for the values* — one fewer field, but clobbers the enclosing definition's index-addressed props when the extension renders its caller content; a real correctness bug. Rejected.
  - *A per-invocation wrapper object bundling map + values* — one field, but allocates on every render even for all-constant sites (violates the zero-per-render-alloc target, D-Perf). Rejected in favour of two carried references.
  - *An accessor on the extension base class instead of `Scope`* — would keep the map off `Scope`, but the plan pins the accessor to `Scope` (P8-Q2) alongside the existing read surface, and the base class cannot intercept the user's own `ProcessData` override to install the frame. Rejected.
- **Grounding.** [custom-extensions → The Scope](../../custom-extensions.md#the-scope); `Scope.WithProps`/`SlotCarrier` precedent; plan P8-Q2; [D6](../common/cross-cutting-decisions.md#d6--the-scope-publishread-channel-is-public-api) (Scope is public extension API).

### D3 — Required vs optional derivation (closes P8-Q3, decided)

- **Decision.** A parameter is **optional** iff `Default != null` **or** `Optional == true`; otherwise it is **required**. When optional, its default value is `Default` (which is `null` in the `Optional`-only case). This reconstructs the definition's `(HasDefault, DefaultValue)`: `HasDefault = optional`, `DefaultValue = Default`. A required parameter unbound at a call site is `HED5002` (`MissingRequiredProp`); an optional parameter absent binds its default — **exactly** the definition-prop rule.
- **Rationale.** Attribute named arguments cannot distinguish "unset" from "default-valued", so a non-null `Default` is the concise, detectable optional marker (covers the common `Default = 3` / `Default = "plain"` cases), and `Optional = true` covers the one residual case a non-null default cannot express — an optional reference/nullable parameter defaulting to null. This mirrors definition semantics with no ambiguity.
- **Alternatives rejected.** *A `Required` boolean defaulting true* — `Required = true` as the default cannot be distinguished from unset, and forcing every optional parameter to also write `Required = false` is verbose against the concise definition form. Rejected. *Inferring an optional value-type default from the type's zero* — definitions require an explicit default for optional value-type props (`compact: bool = false`); inferring `0`/`false` would diverge. Rejected.
- **Grounding.** [language-reference → Props](../../language-reference.md#props-nameprop-type--default); `PropLayout.ApplyDefault`; plan P8-Q3.

### D4 — One prop contract, a new population source across four layers (SD-B)

- **Decision.** The extension's `[Prop]` attributes are a **new population source** for the one prop contract, realized per layer exactly as props already are:
  - **Dynamic runtime:** `PropLayout.ResolveFromExtension` builds the shared internal `PropLayout` from attributes; `HeddleCompiler.BindProps` (refactored to bind against a layout + owner display name rather than a `DefinitionItem`, [WI4](#wi4)) produces the `PropsBinder`, emitting `HED5001`–`HED5004`; a new internal carrier installs the values.
  - **LSP:** transitive through `HeddleCompiler.Compile` (no LSP change).
  - **Generator:** a **parallel new attribute-reading path** — `ResolveExtensionPropLayout` over Roslyn symbols — feeding the **existing** `TryBuildPropsPrototype`; it emits the frozen `object[]` + `PrecompiledPropSetter[]` and binds the carrier via a new `PrecompiledRuntime.BindExtension`.
  - **Precompiled runtime:** a new `PrecompiledRuntime.BindExtension` constructs the engine carrier with the generator-emitted prototype/setters/map (mirroring `BindDefinition`).
  - **Carrier attribute-transparency (encoding correctness — security-sensitive).** The `ExtensionParameterCarrier` is a decorator, but the compiler resolves the **output render type** (`[EncodeOutput]`/`[NotEncode]` → `RenderType.Encode`/`Raw`) and the HED2003 redundant-encode warning from the **concrete instance type** (Assumed state — `InitializeTemplate`/`WarnOnRedundantEncoding` read `GetType()`). The carrier is a plain `AbstractExtension` subclass and carries **neither** attribute, so it must never be the type those reads observe: (a) on the **dynamic tier** the render type is resolved on the **inner** extension — the carrier is installed **after** `InitializeTemplate` ([WI4](#wi4--the-parameter-carrier--dynamic-binding)), so `SetUpRenderType(Encode)` lands on the inner; (b) `WarnOnRedundantEncoding` unwraps the carrier to its inner before reflecting ([WI4](#wi4--the-parameter-carrier--dynamic-binding)); and (c) on the **precompiled tier** the generator computes the same render type from the inner symbol's `[EncodeOutput]`/`[NotEncode]` (the symbolic mirror of `InitializeTemplate`, [WI5](#wi5--generator-extension-parameter-layout--emission)) and `PrecompiledRuntime.BindExtension` applies it to the **inner** via `BindPrecompiled` ([WI6](#wi6--precompiledruntimebindextension)) — **not** hard-coded `Raw`. Adding `[Prop]` to a `[EncodeOutput]` extension therefore changes **neither** its output bytes **nor** its HED2003 surface, and the two backends stay byte-identical on the `EncodedGrid` fixture (the differential harness gates it). Per [coding-standards](../common/coding-standards.md) SOLID precedence (sandbox/security beats ergonomics), this correctness outranks a uniform, one-site wrap.

  The rollout is one wave (all four layers together), so the dynamic and precompiled backends stay at parity from day one; the differential harness gates dynamic == precompiled byte-identity.
- **Rationale.** The prop path already solved typed named binding, positioned diagnostics, and dual-backend carriage; adding a source reuses it (DRY, second consumer of the contract — the differential harness is the standing guard). The generator genuinely cannot link the internal `PropLayout`, so its parallel path is the honest, already-established Phase-7 shape (the generator carries its own parallel tracking), not a table-free freebie.
- **Alternatives rejected.** *A bespoke extension-parameter table/binder/diagnostics* — duplicated surface the prop path already provides; rejected by the plan. *Overloading the positional model* — untyped, unnamed, unlike the definition contract it must mirror; rejected. *Reusing the `[ScopeChannel]` publish/read channel* — sibling coordination, not a caller→callee typed input; rejected.
- **Grounding.** plan *Design direction*; [`PropLayout.cs`](../../../src/Heddle/Runtime/Expressions/PropLayout.cs)/[`PropsBinder.cs`](../../../src/Heddle/Runtime/PropsBinder.cs) (`internal sealed`); [`TemplateEmitter`](../../../src/Heddle.Generator/Emit/TemplateEmitter.cs) `PropLayoutInfo`; [D1](../common/cross-cutting-decisions.md#d1--stable-diagnostic-ids-hedxxxx).

### D5 — The `HED5005` relaxation and reuse of `HED5001`–`HED5004` (SD-C)

- **Decision.** At the sole `HED5005` site (`HeddleCompiler.CompileItem`, the `definitionItem == null` named-argument gate, ≈762–768), the check becomes:

  > If the call carries named arguments and the target is not a definition, resolve the extension type by name. If that extension **declares one or more `[Prop]` parameters**, do **not** error and do **not** abort — let compilation proceed to `CreateExtension`, which binds the named arguments against the extension's `[Prop]` layout (reusing `BindProps`, emitting `HED5001`/`HED5003`/`HED5004` per argument and `HED5002` for any unbound required parameter). Otherwise (the extension declares no parameters, or the name is a function/unknown) raise `HED5005` — same id, same positioned failure, reworded message.

  The gate is **"declares a parameter surface"** (≥ 1 `[Prop]`), identical to how a definition *with* props reaches `HED5001` for an unknown name while a *non-definition* gets `HED5005`. An unknown parameter name on a parameter-declaring extension is therefore `HED5001` (not `HED5005`), exactly like a definition.
- **Reworded `HED5005` message.**
  > `'{name}' declares no named parameters — named arguments can only be passed to a definition that declares props or to an extension that declares [Prop] parameters.`

  Same id, same condition (target exposes no named-parameter surface); only the wording changes (the old text's "is not a definition" is no longer literally the reason).
- **Resolution helper.** Add `internal static bool TemplateFactory.TryGetExtensionType(string, out Type)` — **already present** (≈166) — to obtain the CLR type at the gate; a small `internal static bool PropLayout.DeclaresExtensionParameters(Type)` (true iff the type or a base carries `[Prop]`) answers the gate without building the full layout. The full layout runs once in `CreateExtension`; per-compile caching keyed by extension `Type` avoids duplicate work across call sites (mirroring `ResolveLayoutCached`'s definition-identity cache). Because the layout — and with it the **malformed-declaration** diagnostics `HED5007`/`HED5008`/`HED5009`/`HED5010`/`HED5015` that `ResolveFromExtension` re-raises — is resolved **once per extension `Type` per compile**, those declaration diagnostics are emitted **once**, positioned at the **first call site** of that type in the compile (a representative position; declaration faults are per-*type*, not per-*call* — exactly as a definition's `HED5009`/`HED5010` surface once through `ResolveLayoutCached`). This is the position `ownerCallPosition` carries into `ResolveFromExtension` on the cache-miss (F11). The **per-argument** diagnostics (`HED5001`/`HED5003`/`HED5004` per named argument, `HED5002` per unbound required parameter) are **not** cached: `BindProps` runs at **every** call site, so a mistyped/unknown/duplicate/missing argument is diagnosed at each call, positioned at that call's argument — matching definition-prop behavior.
- **Post-relaxation control flow (verified).** Once the gate at ≈762 stops aborting, a parameter-declaring extension call with named args falls through to the standalone name-resolution block (`HeddleCompiler.CompileItem` ≈789–819) that resolves definition → extension → registered function. That block is benign for this surface: `nameIsExtension` (`TemplateFactory.Exists`) is **true** for a resolved extension, so (a) the `!nameIsExtension && nameInRegistry` branch — the one that raises `FunctionRequiresExpressionArguments` and `return null`s on a non-`functionCompatibleShape` call (a `PropArguments`-carrying call is not function-compatible) — is **unreachable**; and (b) a name that is **both** an extension and a registered function draws only the pre-existing `FunctionShadowedByExtension` **warning**, then proceeds to `CreateExtension` and binds the `[Prop]` layout normally. A function-only name never reaches this block with named args (it fails `DeclaresExtensionParameters` at the gate and takes `HED5005`). So the relaxation introduces no new control-flow interaction beyond the existing shadow warning.
- **Rationale.** The relaxation only lets a **previously-erroring** shape (named args on an extension — a hard `HED5005` today) compile, and only when the extension declares a matching parameter surface — it cannot change any template valid today (back-compat, D-BC). Reusing `HED5001`–`HED5004` gives extensions the identical positioned diagnostics definitions have, for zero new call-time id.
- **Alternatives rejected.** *Gate on the exact name matching a declared parameter* — would make an unknown name `HED5005` instead of `HED5001`, diverging from definitions (a definition with props gives `HED5001` for an unknown name). Rejected — the deliverable is call-site symmetry. *A new call-time id for extensions* — the plan forbids `HED5019+`; the prop family already fits. Rejected.
- **Grounding.** `HeddleCompiler.CompileItem` (≈762); `BindProps` (≈1290); plan *Back-compat*; [diagnostic-ids](../../plan/common/diagnostic-ids.md).

### D6 — Malformed-`[Prop]` diagnostics on both tiers, and `HED7017` (closes P8-Q4)

- **Decision — condition → id mapping.** A malformed `[Prop]` declaration is diagnosed on **both** tiers, never silently:

  | Malformed condition | Dynamic tier (`HeddleDiagnosticIds`) | Generator tier |
  |---|---|---|
  | Two `[Prop]` on one class share a name | `HED5007` (`DuplicatePropDeclaration`) | `HED7017` |
  | Parameter name is reserved (`out` / `this`) | `HED5015` (`ReservedPropName`) | `HED7017` |
  | `Default` not convertible to the parameter type | `HED5009` (`PropDefaultNotConvertible`) | `HED7017` |
  | Parameter `Type` is `null`, an open generic, or a pointer/by-ref (not a usable closed type) | `HED5010` (`UnresolvedPropType`) | `HED7017` |
  | A subclass extension re-declares an **inherited** `[Prop]` with a type **not assignable to** the base's declared type (a widening re-declaration) | `HED5008` (`PropRedeclarationMismatch`) | `HED7017` |

  The five dynamic ids are **re-raised** by the new attribute reader (`PropLayout.ResolveFromExtension` for `HED5010`/`HED5009`/`HED5008`; the reader's pre-checks for `HED5007`/`HED5015`), because their existing raise sites (`ParseContext` for `HED5007`/`HED5015`; `PropLayout.Resolve` over `PropDeclaration` for `HED5008`/`HED5009`/`HED5010`) only run over definition headers. `HED5008` reaches the extension surface because `[Prop]` is `Inherited = true` (D1): a subclass extension inherits its base's `[Prop]`s and may re-declare one, so the shared `byName` re-declaration branch WI2 reuses (`PropLayout.cs` ≈104–118, the `!existing.Type.Type.IsType(type.Type)` assignability check) is reachable — an assignable narrowing re-defaults the base slot; a non-assignable widening raises `HED5008`. It keeps its definition meaning unchanged; its generator twin is `HED7017` (the re-declaration-widening fault, detected symbolically in `ResolveExtensionPropLayout` — WI5). On the dynamic tier these are positioned at the **extension call site** (`extensionItem.Position`) — the extension's attribute source is in (possibly referenced) C#, with no template position; the call site is the same anchor `HED7015`/`HED7006` use for referenced-assembly extensions.

  > **CORRECTION (recorded, with evidence).** With the ratified `typeof()` type argument (D1), an *unresolved-type-name* condition **cannot arise** — the C# compiler rejects `typeof(Unknown)` before Heddle sees it. `HED5010`/`HED7017`-for-type therefore fire only on the residual **unusable-type** conditions above (a `null` type argument, or an open-generic/pointer/by-ref `typeof`), which remain reachable. `HED5010` keeps its definition meaning unchanged. This narrows the plan's "unresolvable type" wording to the conditions the chosen API actually admits.
- **Decision — `HED7017` (claimed into `GeneratorDiagnostics` + the registry).**
  - **Id / title / category / severity:** `"HED7017"` / `"Malformed extension parameter"` / `"Heddle.Precompile"` / `DiagnosticSeverity.Error`, `isEnabledByDefault: true` (a `DiagnosticDescriptor` in `GeneratorDiagnostics`, **not** a `HeddleDiagnosticIds` constant — same posture as `HED7016`).
  - **Message:** `Extension <{0}> declares a malformed [Prop] parameter: {1}` where `{0}` is the extension name and `{1}` names the specific fault (`duplicate parameter name '<n>'` / `reserved parameter name '<n>'` / `default value for '<n>' is not convertible to <type>` / `parameter '<n>' has an unusable type` / `re-declared parameter '<n>' widens the inherited type <inheritedType>`).
  - **Trigger:** build time, when `ResolveExtensionPropLayout` detects any malformed condition for an extension a template **binds** (a bodiless parameter-declaring call). It fires regardless of the call's named arguments (a malformed layout corrupts even a defaults-only bind).
  - **Position:** the extension call site in the template (the generator has no template location for the referenced-assembly attribute).
  - **Severity justification / additivity:** `HED7017` fires **only** on the new `[Prop]` surface; no extension can carry a malformed `[Prop]` today (the attribute is new), so it touches no template valid today. Error severity matches the dynamic-tier compile errors it twins.
- **Rationale.** The generator does not run `HeddleCompiler` and forwards only parse-layer diagnostics (Assumed state), so the dynamic-tier compile ids (`HED5009`/`HED5010`) never reach a build; without `HED7017` a malformed declaration would degrade **silently** on the build tier (`layout.Failed`). A dynamic-only host, conversely, has no `HED7xxx`. Two twins, one per tier, close the gap the plan names.
- **Alternatives rejected.** *Forward the dynamic ids from the generator* — the generator has no `HeddleCompiler` pass over the extension's reflected attributes and cannot reproduce it from metadata cheaply; the parallel path is the established shape. Rejected. *Four distinct `HED70xx` ids* — the plan pins a single next-free `HED7017`; the message discriminates the fault. Rejected. *Silent fallback (`layout.Failed`) with no id* — violates the "never silently" success criterion. Rejected.
- **Grounding.** [`GeneratorDiagnostics`](../../../src/Heddle.Generator/Diagnostics/GeneratorDiagnostics.cs) (`HED7016` highest); `HeddleTemplateGenerator.ParseAndReport`; plan P8-Q4; [diagnostic-ids](../../plan/common/diagnostic-ids.md).

### D7 — Precompiled: bodiless binds at build, bodied reuses the existing fallback (verified)

- **Decision.** A **bodiless** parameter-declaring call (`@grid(Photos, columns: 4)`) binds its named arguments **at build time** on the precompiled tier — reusing `TryBuildPropsPrototype`'s frozen prototype + `PrecompiledPropSetter[]`, installed through the new `PrecompiledRuntime.BindExtension` — full parity with a bodiless definition-with-props call. A **bodied** parameter-declaring call (`@grid(x, columns: 4){{…}}`) follows the **existing** bodied-custom-call contract: it **falls back to the dynamic tier** (`BuildCustomExtensionCall`, verified ≈645–648), where byte-parity is preserved *by* the fallback (the dynamic tier renders it, parameters and all). There is **no new degrade-to-dynamic path**.
- **Rationale.** Bodied custom calls already fall back (their body model-typing is extension-specific); this phase changes nothing there. Bodiless is the build-time parity path props already established.
- **Grounding.** [custom-extensions → Precompiled mode](../../custom-extensions.md#precompiled-mode); `BuildCustomExtensionCall` (≈645); `PrecompiledRuntime.BindDefinition` precedent.

### D8 — Plain-custom extension render type derives from `[EncodeOutput]`/`[NotEncode]` (resolves P8-J-E1)

- **Status — maintainer ruling (2026-07-15).** The maintainer chose to **fix, in this phase, the pre-existing cross-tier divergence P8-J-E1** rather than escalate it as an out-of-scope item. This supersedes the earlier review disposition (P8-J-E1 as "the only open, non-blocking escalation") that had carved the plain-custom path out as pre-existing and out-of-scope; [Back-compat](#back-compat-and-migration) and the Testing plan now record it as **resolved in-scope**.
- **The divergence (verified against source).** The dynamic tier resolves a custom extension's output render type in [`HeddleCompiler`](../../../src/Heddle/Runtime/HeddleCompiler.cs) `InitializeTemplate` (≈1460) from the **concrete extension type**: `extension.GetType().IsHaveAttribute<EncodeOutputAttribute>(true) ? (IsHaveAttribute<NotEncodeAttribute>(true) ? RenderType.Raw : RenderType.Encode) : RenderType.Raw`, then `SetUpRenderType`. The generator's **parameterized** path already mirrors this (WI5's `AllocateParameterizedExtension` emits the same expression from `Info.HasEncodeOutput`/`Info.HasNotEncode`, H1). But the generator's **non-parameterized** path — [`TemplateEmitter.AllocateCustomExtension`](../../../src/Heddle.Generator/Emit/TemplateEmitter.cs) (≈658) — **hard-codes** `global::Heddle.Data.RenderType.Raw` in its `PrecompiledRuntime.Bind(...)` emission (≈665, verified). So a **no-parameter** custom extension marked `[EncodeOutput]` renders **`Raw` on the precompiled tier** while the **dynamic tier resolves `Encode`** — a silent, byte-changing cross-tier divergence (an HTML-encoding/**XSS-class** hazard) that exists **today, independent of parameters**.
- **Decision.** The generator's non-parameterized allocation `AllocateCustomExtension` **derives** the render type from the inner extension's `[EncodeOutput]`/`[NotEncode]` symbols — `Info.HasEncodeOutput ? (Info.HasNotEncode ? RenderType.Raw : RenderType.Encode) : RenderType.Raw`, emitted as the `global::Heddle.Data.RenderType.*` literal — the **exact** expression `InitializeTemplate` evaluates on the dynamic tier and the **same** expression WI5's parameterized path emits. It **must not** stay hard-coded `Raw`. `PrecompiledRuntime.Bind` already applies its `renderType` argument to the extension via `BindPrecompiled` (`PrecompiledRuntime.cs` ≈34), so the derived type lands `DirectRender` on the extension exactly as the dynamic tier's `SetUpRenderType` does. Realized as [WI5b](#wi5b); it reuses the `Info.HasEncodeOutput`/`Info.HasNotEncode` flags WI5 adds to [`ExtensionBinder.Info`](../../../src/Heddle.Generator/Emit/ExtensionBinder.cs) (the symbolic mirror of `IsHaveAttribute<…>(true)`, both attributes surfaced over the base-type chain via the existing `HasAttribute` helper) — no new generator machinery.
- **Byte-identity gate holds.** The change affects **only** the previously-divergent case — a no-parameter `[EncodeOutput]` (not also `[NotEncode]`) custom extension that **reaches `AllocateCustomExtension`** on the precompiled tier — aligning precompiled **to** the dynamic tier. Two seam populations share `AllocateCustomExtension` and must both be accounted for; neither moves an existing golden:
  - **Test/user-fixture `[EncodeOutput]` extensions.** Had one existed in the differential corpus it would render `Raw` on precompiled vs `Encode` on dynamic, so the differential would be **failing today** (`Raw != Encode`); the green suite proves none exists (verified: no `[EncodeOutput]` appears in `Heddle.Generator.IntegrationTests/Fixtures/*`).
  - **Referenced-assembly (engine built-in) `[EncodeOutput]` extensions.** `AllocateCustomExtension` is also reached for any name `_extensionBinder.TryResolve` returns, which includes the engine built-ins `@date/@money/@string/@url/@int/@time/@attr` and `@html` (all `[EncodeOutput]`; `@guid` is verified **not** `[EncodeOutput]` — [`GuidExtension.cs`](../../../src/Heddle/Extensions/GuidExtension.cs) ≈36 — so it is not in this set) — so the `Fixtures/*` grep alone does **not** cover them. Seven of the eight (`@date/@money/@string/@url/@int/@time/@attr`) override `InitStart`, so `info.OverridesHook == true` routes them to the **dynamic fallback** (`TemplateEmitter.cs` ≈635–640) **before** `AllocateCustomExtension` — WI5b never touches them. The eighth, `@html` ([`EmptyHtmlExtension`](../../../src/Heddle/Extensions/EmptyHtmlExtension.cs), an `AbstractHtmlExtension` that does **not** override `InitStart`/`CompleteInit`), **does** reach the seam when called **bodiless**; WI5b changes its emitted render type from hard-coded `Raw` to derived `Encode`, which **aligns** it to the dynamic tier (which already resolves `Encode`). This is not a regression: the corpus contains **no** bodiless `@html(…)` custom-call fixture (verified: 0 grep hits in `Heddle.Generator.IntegrationTests`), and by the same "differential would already be failing" argument any existing bodiless `@html` producing markup-significant bytes would already be red — non-markup output is byte-identical under `Raw` and `Encode`. The `Fixtures/*` grep is therefore sufficient **only in combination with** the `OverridesHook` → dynamic-fallback invariant (covering the seven formatters) and the verified absence of a bodiless `@html` call reaching the seam.

  A plain custom extension **without** `[EncodeOutput]` still emits `Raw` (the expression's else-branch), byte-unchanged. (The `[NotEncode]` term of the expression is a **structurally dead** branch on both tiers, retained only to mirror `InitializeTemplate` verbatim: `NotEncodeAttribute` is `[AttributeUsage(AttributeTargets.Property)]` ([`NotEncodeAttribute.cs`](../../../src/Heddle/Attributes/NotEncodeAttribute.cs)), so no extension **class** can carry it — `IsHaveAttribute<NotEncodeAttribute>(true)` on the dynamic tier and the generator's `HasAttribute(type, NotEncodeAttribute)` over the class base-chain are **both always false**; the two dead branches agree, introducing no divergence.) The new `EncodedBare` differential fixture (WI9) is the standing guard proving the tiers now agree.
- **Rationale.** Both tiers must resolve one custom extension's render type identically; a hard-coded `Raw` on one tier is a latent XSS-class bug the differential harness would have caught only once a fixture exercised it. Deriving the type from the same attributes the dynamic tier reads is the minimal correct fix, symmetric with the parameterized path (H1) — one seam, one expression, both tiers.
- **Alternatives rejected.** *Leave it out-of-scope / escalate only* — the maintainer ruled it in-scope (2026-07-15); a known silent cross-tier encoding divergence should not ship unfixed beside the parameterized path that already resolves the identical concern. Rejected. *Fix it in `PrecompiledRuntime.Bind` (runtime) instead of the emitter* — `Bind` faithfully applies whatever render type it is handed; the defect is the **caller** passing a constant, so the emitter is the correct seam (and the runtime `Bind` stays a general primitive). Rejected. *Route no-parameter custom extensions through the parameterized `AllocateParameterizedExtension` path* — that path carries the parameter carrier/prototype machinery a parameter-less extension has no use for (needless allocation and a behavioral change beyond the render type); the one-line render-type derivation in `AllocateCustomExtension` is the surgical fix. Rejected.
- **Grounding.** `InitializeTemplate` (≈1460) / `AllocateCustomExtension` (≈658–666) / `PrecompiledRuntime.Bind` (≈28–36), all verified; WI5's `Info.HasEncodeOutput`/`Info.HasNotEncode`; [D4](#d4--one-prop-contract-a-new-population-source-across-four-layers-sd-b) carrier-transparency (the parameterized twin); P8-J-E1 (review finding, [OQ6](OPEN-QUESTIONS.md)).

Accept the re-declaration **iff**:

    ```csharp
    var conv = Compilation.ClassifyConversion(reDeclaredType, inheritedType);
    bool srcNullable = reDeclaredType is INamedTypeSymbol { OriginalDefinition.SpecialType: SpecialType.System_Nullable_T };
    bool dstNullable = inheritedType  is INamedTypeSymbol { OriginalDefinition.SpecialType: SpecialType.System_Nullable_T };
    bool accept =
        // (A) reflection's underlying-value rule: X is assignable to X?  (int -> int?)
        (dstNullable && SymbolEqualityComparer.Default.Equals(((INamedTypeSymbol)inheritedType).TypeArguments[0], reDeclaredType))
        // (B) identity + implicit reference (class/interface upcast, array & generic variance)
        || conv.IsIdentity
        || (conv.IsImplicit && conv.IsReference)
        // (C) implicit boxing (value -> object / ValueType / implemented interface) — EXCEPT a
        //     Nullable<T> source to an interface: reflection treats Nullable<T> as implementing NO
        //     interfaces, so int? -> IComparable is NOT assignable although Roslyn calls it boxing.
        || (conv.IsImplicit && conv.IsBoxing && !(srcNullable && inheritedType.TypeKind == TypeKind.Interface));
    ```

    This equals `inheritedType.IsAssignableFrom(reDeclaredType)` over the **whole reachable declared-prop domain** — reference upcast, identity, boxing, array & generic variance, **and `Nullable<T>` in both directions** — closing the two nullable cracks a bare `identity|reference|boxing` (or `.IsImplicit`) predicate leaves open:
    - **`int?`←`int`** (rule `(A)`): runtime `IsAssignableFrom(int?, int)` = **True** → accepted as an assignable narrowing (re-defaults the base slot). Roslyn classifies `int`→`int?` as `ImplicitNullable` (neither reference nor boxing), so a bare `identity|reference|boxing` predicate would **false-error** (`HED7017` on a template the dynamic tier renders); rule `(A)` restores the accept.
    - **`IComparable`←`int?`** (the `(C)` interface exclusion): runtime `IsAssignableFrom(IComparable, int?)` = **False** (`Nullable<int>` the type implements no interfaces) → **`HED7017`**. Roslyn classifies `int?`→`IComparable` as **boxing**, so an unguarded `.IsBoxing` would **silently accept** — the exact silent cross-tier divergence this twin exists to close (P8-J-H2). The `srcNullable && interface` exclusion eliminates it absolutely.
    - **`int?`←`object`/`ValueType`** stays accepted via `(C)` (target is not an interface — reflection assigns a boxed nullable up its base chain, `object.IsAssignableFrom(int?)` = True); **`int`←`int?`** (`int.IsAssignableFrom(int?)` = False), **`long`←`int`** (numeric), and every user-defined / explicit conversion stay **rejected** — matching reflection exactly.

## Back-compat and migration

Governed by [R5](../../plan/common/standing-rulings.md#standing-rulings-user-set) (additive) and the byte-identity gate — **no** breaking-window change; purely additive. Proof of additivity:

- **The relaxation only makes a previously-erroring shape compile.** Passing a named argument to an extension is a hard `HED5005` **today** (Assumed state, `HeddleCompiler.CompileItem` ≈762), so no template that compiles today exercises the relaxed path. `@grid(x, columns: 4)` starts compiling **only** when `grid` declares a `columns` parameter — it cannot change any template valid today (the `@@`-escape shape of additivity).
- **Parameter-less extensions are unchanged.** An extension declaring no `[Prop]` keeps `HED5005` on the same condition — same id, same positioned failure (message reworded only). Covered by the parameter-less validation row.
- **Definition props unchanged.** `HED5005`/`HED5006` keep their exact meaning for definitions; `HED5001`–`HED5004` keep their meaning and are simply now **reachable** for a parameter-declaring extension. `HED5006` never reaches the extension surface (short-circuit).
- **Rendered output byte-identical.** No template valid today changes its bytes; the dynamic == precompiled differential does not move on any existing fixture (regression gate), and new parameter fixtures render identically under both backends.
- **New `Scope` fields are internal and default-null.** A prop-less / parameter-less execution carries `null` for both new fields (the common path); no existing render allocates or behaves differently.
- **Encoding behavior is preserved (F6), on both tiers.** The parameter carrier is attribute-transparent: on the dynamic tier the render type is resolved on the inner extension (the carrier is installed *after* `InitializeTemplate`) and `WarnOnRedundantEncoding` unwraps it; on the precompiled tier the generator computes the inner's render type from its `[EncodeOutput]`/`[NotEncode]` symbols (WI5) and `BindExtension` applies it to the inner (WI6). Adding `[Prop]` to a `[EncodeOutput]`/`[NotEncode]` extension therefore changes neither its output bytes nor its `HED2003` surface, and the two backends stay byte-identical on the `EncodedGrid` fixture (in the differential harness, H1). A wrap-*before* install (dynamic) or a hard-coded `RenderType.Raw` (precompiled) would have silently flipped such an extension to non-encoding (an XSS-class regression); the wrap-*after* rule and the computed precompiled render type foreclose both, guarded by the F6 cross-tier differential.
- **Resolved in-scope: plain-custom `[EncodeOutput]` tier divergence (P8-J-E1).** The generator's **pre-existing** `AllocateCustomExtension` path (a no-parameter custom extension) hard-coded `RenderType.Raw` (`TemplateEmitter.cs` ≈665), so a plain `[EncodeOutput]` custom extension rendered **encoded on the dynamic tier and raw on the precompiled tier** — a latent, byte-changing cross-tier (XSS-class) divergence independent of parameters. The maintainer ruled this **in-scope** (2026-07-15); [WI5b](#wi5b)/[D8](#d8--plain-custom-extension-render-type-derives-from-encodeoutputnotencode-resolves-p8-j-e1) now derive the plain-custom render type from `[EncodeOutput]`/`[NotEncode]` with the **same** expression the dynamic tier and the parameterized path use, so both tiers agree. **The byte-identity gate holds:** the fix moves bytes **only** in the previously-divergent case (a precompiled no-parameter `[EncodeOutput]` extension **reaching `AllocateCustomExtension`**), aligning precompiled **to** the dynamic tier's already-correct encoding. It moves **no** existing corpus fixture across **both** seam populations (see the [D8](#d8--plain-custom-extension-render-type-derives-from-encodeoutputnotencode-resolves-p8-j-e1) byte-identity gate): (i) no **test/user-fixture** is a plain `[EncodeOutput]` custom extension (had one been, the differential would already be failing); and (ii) among the **engine built-in** `[EncodeOutput]` extensions that share the seam, the seven formatters (`@date/@money/@string/@url/@int/@time/@attr`) override `InitStart` → dynamic fallback **before** the seam (untouched), and the eighth, `@html` ([`EmptyHtmlExtension`](../../../src/Heddle/Extensions/EmptyHtmlExtension.cs), no hook override), reaches the seam only on a **bodiless** call, of which the corpus contains none (verified). (`@guid` is **not** `[EncodeOutput]` — [`GuidExtension.cs`](../../../src/Heddle/Extensions/GuidExtension.cs) ≈36 — so it never enters this population.) Every non-`[EncodeOutput]` plain-custom extension stays byte-identical (still `Raw`). The `EncodedBare` differential fixture (WI9), pinned to provably hit the seam and non-vacuous against the pre-fix `Raw`, guards the alignment.
- **No new tier divergence for parameter-less named-arg calls (F7).** A bodiless custom call carrying named args to an extension that declares no matching `[Prop]` now **degrades** on the precompiled tier so the dynamic tier's `HED5005` governs both tiers. This **closes** a pre-existing latent divergence (the generator would otherwise bind the call with the named args silently dropped) rather than widening one; no template valid today is affected (named args on a parameter-less extension are a hard `HED5005` today).
- **No migration note required** (nothing breaks). Docs edits ([custom-extensions](../../custom-extensions.md), [language-reference Props](../../language-reference.md#props-nameprop-type--default)) are downstream, additive.

## Performance considerations

- **Parameter binding is compile/build-time.** The `PropLayout` (dynamic) / `PropLayoutInfo` (generator) resolution, the `PropsBinder`/prototype construction, and the `ExtensionParameterMap` build all happen once per call site at compile/build — never per render. The map is cached per extension `Type`; the layout binding is memoized per call site.
- **Render-time read is O(1), no added per-render allocation.** `TryGetParameter`/`GetParameter` is one ordinal `Dictionary` lookup + one array index. For an **all-constant** parameter set the carrier installs the **shared frozen** values array (zero alloc, identical to `PropsBinder.IsAllConstant`); a **dynamic** argument incurs the **same single per-invocation clone** definition props already do (`PropsBinder.Bind` / precompiled `Clone`) — the extension path introduces **no** allocation beyond the prop contract's existing cost. `WithExtensionParameters` returns a `readonly struct` copy (no heap).
- **Parameter-less extensions pay nothing.** No `[Prop]` → no carrier wrap, no frame, both new `Scope` fields `null` — allocation-identical to today.
- **Guarding benchmark.** The existing prop/differential allocation posture guards this (no new hot path); the all-constant zero-alloc claim is asserted by the frozen-array identity check the props suite already carries, extended to a parameter fixture.

## Standards compliance

- **DRY — a second consumer of the prop contract.** The binding table, binder, positioned diagnostics, and dual-backend carriage are **reused**, not duplicated; the extension `[Prop]` attributes are a new *population source*, exactly as Phase 7's content-regions are a second consumer of their machinery. The differential harness is the standing guard that the two backends stay identical.
- **YAGNI.** No bespoke extension-parameter table/binder/diagnostics; no typed generic accessor (the untyped `object` read mirrors `ModelData` — authors already cast); `Optional` exists only because the one null-default case genuinely needs it. The render carriage is name-addressable **only** because an opaque C# extension intrinsically reads by name — the minimal deviation from the index-addressed definition path, justified by a real difference, not gold-plating.
- **API design / compat.** Additive public surface (one attribute, two `Scope` methods, one generated-code entry); `netstandard2.0`-clean (no BCL types beyond `Dictionary`/`Type`); the carrier/map stay `internal` (the generated code reaches them only through the public `BindExtension`, the established internal-access posture). No per-render state on extension instances — all per-render state rides the `Scope` lineage ([coding-standards](../common/coding-standards.md) thread-safety rule).
- **Grammar stability.** **No** grammar change — the named-argument production is reused verbatim (plan pin verified against `CompileItem`, which already parses `CallParameter.PropArguments` for any target).

## Deferred items

| Item | Trigger |
|---|---|
| Content regions / slots on extensions | Explicitly rejected and out (an extension has no template body); a future need for extension *content* would be new work with its own design. |
| A typed generic accessor (`GetParameter<T>`) | An author-demand signal that the `object`-cast read is ergonomically insufficient; the untyped read matches `ModelData` today. |
| `[Prop]` on built-in extensions | Built-ins are unchanged this phase; adding a parameter to a built-in verb would revisit its declared surface. |
| Precompiled build-time bind of a **bodied** parameter-declaring call | Only if the general bodied-custom-call fallback is ever lifted (a separate, larger effort); today bodied custom calls fall back to the dynamic tier by design, and this phase reuses that. |
| Optional-null default expressed without `Optional = true` | Only if attribute metadata ever gains an unset/null distinction; today `Optional = true` covers it. |
