## Scope and goal

Make the already-public, already-documented `EnableFileChangeCheck` / `OnFileChanged` / `OnFileDeleted` / `OnFileRenamed` contract functional for a template compiled from a file. Today the feature is wired but inert, and three compounding defects each independently break it (all verified against source in [Assumed state](#assumed-state)):

1. The `FileSystemWatcher.Filter` is built from `TemplateName` **without** `FileNamePostfix`, so it never matches the real on-disk file the reader reads.
2. `FileSystemWatcher.EnableRaisingEvents` is never set `true`, so the watcher raises no events even with a correct filter.
3. A recompile reuses the same already-`Compiled` `CompileScope`, so delayed-extension finalization (`CompleteInit`) and the `FullCSharp` Roslyn pass are skipped on the 2nd and later reloads — the document served is stale/unfinalized.

Per [R6](../../plan/common/standing-rulings.md#standing-rulings-user-set) these are **one deliverable**: fixing a subset would expose a hot reload that silently serves a stale document — worse than today's honest no-op. The deliverable additionally makes the live-document swap safe under concurrent renders (publish a coherent old-or-new document; release the superseded document through Phase 4's deferred-dispose discipline instead of orphaning it as the current code does).

**In scope:** the four seams above plus the two downstream doc reconciliations the plan names ([csharp-api.md](../../csharp-api.md#file-watching) "Current limitation"; [language-assessment.md](../../language-assessment.md) §4.6 / §8).

**Out of scope** (plan non-goals): async recompile (recompile stays synchronous on the watcher callback); a general hot-reload / file-provider abstraction; watching the transitive `@<<` import closure (root file only — future seam, P1-Q1); the precompiled backend (no watcher); debounce/coalescing (P1-Q5 — recompiles are output-idempotent after the stale-recompile fix; the only residual cost is a per-reload FullCSharp assembly load, D6). No new `HED*` diagnostic (see [Diagnostics](#diagnostics--error-surface)).

## Design decisions

Every P1 plan question, lean, and pin is closed. The single deliverable is realized by decisions D1–D6 (P1-Q2→D4/D5; P1-Q3→D4; P1-Q4→D3–D5; P1-Q5→D6; D3–D5 close the concurrency question P1-Q4). **P1-Q1** (watch the root file only; the transitive `@<<` import closure is a future seam) is closed as a **scope non-goal / deferred item** — see [Scope and goal](#scope-and-goal) and [Deferred items](#deferred-items) — not inside a numbered D, because it is a boundary the deliverable holds rather than a mechanism it builds.

### D1 — Watcher filter = the reader's real file name (defect 1, closes the filter bug)

- **Decision.** Build the watcher's directory and `Filter` from the **same** name [`FileReader`](../../../src/Heddle/FileReader.cs) reads, by reusing the already-constructed `_reader.GetFileName()` (which returns `RootPath / (TemplateName + FileNamePostfix)`), instead of re-deriving a name that drops the postfix:

  ```csharp
  var watchedFile = _reader.GetFileName();                 // e.g. …/home.heddle — the real read path
  var directory = Path.GetDirectoryName(watchedFile);
  _watchFileName = Path.GetFileName(watchedFile);          // e.g. home.heddle — stored for the rename predicate (D4)
  _watcher = new FileSystemWatcher(directory)
  {
      NotifyFilter = NotifyFilters.FileName | NotifyFilters.Size | NotifyFilters.LastWrite,
      Filter = _watchFileName
  };
  ```

- **Rationale.** The read path and the watch filter are the **same knowledge** (which file this template is); deriving both from one source (`_reader.GetFileName()`) makes them impossible to diverge again (DRY at the load-bearing seam — [coding standards](../common/coding-standards.md#dry-applied)). The `FileReader` ctor guarantees a non-blank postfix on this path, so `_watchFileName` always equals the actual file name.
- **Alternatives rejected.** *Re-derive `Path.Combine(RootPath, TemplateName + FileNamePostfix)` inline at the watcher site* — correct but duplicates the read-path formula a third time (`FileReader.GetFileName`, `TemplateOptions.FullPath`, here), re-opening the exact divergence that caused this bug. *Filter on `TemplateName` and post-filter in the handler* — leaves the OS delivering (or not) events by a name that isn't the file, and the plan's success criterion is explicitly about the installed `Filter` value.
- **Grounding.** [`FileReader.GetFileName`](../../../src/Heddle/FileReader.cs); [`HeddleTemplate.Compile(CompileContext)`](../../../src/Heddle/HeddleTemplate.cs) watcher block.

### D2 — Arm the watcher, exactly when `EnableFileChangeCheck && file-compiled` (defect 2)

- **Decision.** Set `_watcher.EnableRaisingEvents = true;` as the **last** statement of the watcher-construction block, after the handlers are wired. That block is reached **only** from `Compile(CompileContext)` and **only** inside `if (context.Options.EnableFileChangeCheck)` — i.e. exactly when the template is compiled from a file (the file was read by `_reader`) **and** the opt-in flag is set. Inline-string constructors (`HeddleTemplate(string, …)`) and the `!EnableFileChangeCheck` path never construct or arm a watcher. Arming happens regardless of whether the *first* compile succeeded, so that fixing a broken source on disk recovers the template (a subsequent good edit recompiles); a first compile that **threw** (e.g. file-not-found) lands in the existing `catch` (≈277-281) and never reaches the arming statement, so no watcher is armed on a hard read failure.
- **Rationale.** A `FileSystemWatcher` raises nothing until enabled; arming at the end of construction is the minimal, single-line fix. Gating strictly by the code location (`EnableFileChangeCheck` block within the file-compile path) is the precise realization of "only when `EnableFileChangeCheck && file-compiled`" without adding a new predicate. Arming on a soft compile failure (template errors, not an exception) maximizes the dev-loop value (edit-to-fix reloads) at zero back-compat cost, since the flag is opt-in.
- **Alternatives rejected.** *Arm only on first-compile success* — would strand a template whose file currently has a typo, defeating the fix-and-reload loop that is the feature's point. *Arm from a separate public `StartWatching()` call* — a new API surface for a defect fix; the documented contract is "set the flag and it works," not "set the flag then call a method" (YAGNI, [coding standards](../common/coding-standards.md#yagni-applied)).
- **Grounding.** [`HeddleTemplate.Compile(CompileContext)`](../../../src/Heddle/HeddleTemplate.cs); [FileSystemWatcher.EnableRaisingEvents](https://learn.microsoft.com/en-us/dotnet/api/system.io.filesystemwatcher.enableraisingevents).

### D3 — Each recompile compiles into a FRESH context honoring the original options + model type (defect 3, closes the stale-recompile bug)

- **Decision.** The watcher recompile builds a **fresh** `CompileScope` over a **fresh** `CompileContext` reconstructed from the **original** options and root model type, plus the caller-set context state the public ctor does not carry — exactly as the first compile and the public `Recompile` do, but seeding the fields a bare `new CompileContext(options, modelType)` would drop; it does **not** reuse `_context`:

  ```csharp
  CompileResult = Compile(new CompileScope(
      new CompileContext(_watchOptions, _watchModelType)
      {
          ControllerName = _watchControllerName,   // TemplateResolver sets this; the public ctor defaults it to null
      }), document);
  ```

  where `_watchOptions` (a `TemplateOptions`) and `_watchControllerName` (a `string`) are captured **once**, in the watcher-construction block of `Compile(CompileContext)`, from `context.Options` and `context.ControllerName`; `_watchModelType` (an `ExType`) is captured from `context.RootScopeType` **before** the first inner compile (the `Compile(new CompileScope(context), document)` call that runs *ahead of* the watcher block), i.e. its **original ctor value** (`RootScopeType = ScopeType = modelType ?? typeof(object)`, always non-null). It must be captured **pre-compile** because a **root-level `@model`** directive reassigns `RootScopeType` **in place during** the compile: [`ModelExtension.InitStart`](../../../src/Heddle/Extensions/ModelExtension.cs):23–24 sets **both** `initContext.CompileScope.ScopeType` **and** `RootScopeType = modelType`, and the compile-scope setter writes straight through to the root `CompileContext` ([`CompileScope.cs`](../../../src/Heddle/Runtime/CompileScope.cs):30); the root template item is `InitStart`-ed on the root scope ([`HeddleCompiler.cs`](../../../src/Heddle/Runtime/HeddleCompiler.cs) `InitializeTemplate` ≈1464–1468), so a root `@model` mutates the very `context.RootScopeType` the watcher would otherwise capture. Capturing `context.RootScopeType` **after** the compile (in the watcher block) would capture the `@model`-flipped type; a reload that edits or removes the `@model` directive would then seed that **stale directive-flipped** model type — the exact `OutputProfile` divergence class below, applied to the model type (a reload dropping `@model Foo` from a file first compiled under the default `object` model would seed `Foo`, while a from-scratch compile seeds `object`, diverging both the render model-type guard — which reads `ScopeType` — and the FullCSharp `RootModelType`). Capturing the pre-compile original and letting the freshly-parsed `@model` re-apply each reload is from-scratch-equivalent. `RootScopeType` is copied **by value** onto child contexts ([`CompileContext.cs`](../../../src/Heddle/Runtime/CompileContext.cs):109), so only a **root** `@model` affects the captured root value. `ControllerName` is set on the caller's context by [`TemplateResolver`](../../../src/Heddle/Runtime/TemplateResolver.cs) (≈98/103/122/137/140) and is **not** reproduced by `new CompileContext(TemplateOptions, ExType)` (≈142-153, which leaves it `null`), so it must be seeded explicitly or an MVC-resolved reload would silently lose its controller scope. A fresh `CompileContext` has `Compiled == false` and a fresh `CSharpContext` (`CSharpContext.Compiled == false`, empty `Methods`), so `CompleteInit` and the `FullCSharp` Roslyn pass **run every recompile** — the reload produces a fully finalized document equivalent to a first compile of the new content.

  **`OutputProfile` is deliberately *not* captured or seeded** — it is re-derived from `_watchOptions` and any `@profile()` directive in the freshly-parsed source, exactly as a from-scratch compile does. `new CompileContext(_watchOptions, _watchModelType)` sets `OutputProfile = Options.OutputProfile` in its ctor ([`CompileContext.cs`](../../../src/Heddle/Runtime/CompileContext.cs):146); the top-level `@profile()` directive then flips `context.OutputProfile` **during** the recompile ([`ProfileExtension.InitStart`](../../../src/Heddle/Extensions/ProfileExtension.cs):51, `context.OutputProfile = profile` on the root scope's `CompileContext`). Capturing `context.OutputProfile` in the watcher block would capture it **post-compile — after** any `@profile()` mutation, seeding a *stale directive-flipped* profile: e.g. an original `@profile(text)…` file under the default `Options.OutputProfile = Html` captures `Text`; edit the file to **remove** the directive and a from-scratch compile renders HTML-encoded (Options default) while a captured-profile reload would render Text — a divergence from the from-scratch result and from the phase's "honor the original options" criterion. Re-deriving from `Options` + freshly-parsed directives (per [R3](../../plan/common/standing-rulings.md#standing-rulings-user-set) — output context is explicit, from options/directives, never inferred or carried across a reload) is the only from-scratch-equivalent behavior. (Model type is captured **pre-compile** — from the original ctor `RootScopeType`, before any root `@model` runs — because `@model` reassigns `RootScopeType` in place during the compile just as `@profile` mutates `OutputProfile` ([`ModelExtension.cs`](../../../src/Heddle/Extensions/ModelExtension.cs):23–24). Both directive-mutated root fields are reconstructed from their original/derived source rather than captured post-compile: the model type from the pre-compile snapshot, `OutputProfile` from `Options` + the freshly-parsed `@profile`.)

  The mechanism chosen is **compile-into-a-fresh-context**, *not* "reset `Compiled` / `CSharpContext.Compiled` on the reused scope." (The identical reconstruction recipe — fresh `CompileContext` from `Options` + `RootScopeType` + `ControllerName` — is what Phase 4's `TryCompilation(CompileContext)` isolation uses to avoid poisoning the caller's context; the two phases share the pattern.)
- **Rationale.** Resetting only the two `Compiled` flags is **incorrect**, not merely more surgical: a reused `CompileScope` carries `CompiledItems`, `ResolvedPropLayouts`, `CSharpContext.Methods`/`ClassGuid`, and the **never-reset** `CompileErrors`/`CompileWarnings` lists (verified in [Assumed state](#assumed-state)). Re-running `HeddleCompiler.Compile` over new content on that scope would accumulate stale compiled items and methods, regenerate a Roslyn class keyed by the same `ClassGuid` from a mixed method set, and surface the previous reload's diagnostics on the new `CompileResult`. A fresh context is the **only** state that reproduces a first compile; it is also the smallest change (one call site: `_context` → a fresh scope) and it aligns the watcher path with the already-correct public `Recompile` (≈246-250), which is precisely the pattern the buggy watcher path failed to follow. Reconstructing from the original `TemplateOptions` instance (shared, immutable-at-render config) and `RootScopeType` honors every original option — model type, `ExpressionMode`, `OutputProfile`, `Encoder` (by-reference identity preserved), `RenderBudget`, `TrimDirectiveLines`, `Functions`, `MaxRecursionCount` — so nothing but the changed content differs.
- **Alternatives rejected.** *Reset `context.Compiled = false` + `CSharpContext.Compiled = false` before re-running finalize on `_context`* — incorrect (accumulated `CompiledItems`/`Methods`, leaked prior diagnostics, `ClassGuid` collision), and would require additionally clearing five collections and re-nulling per-compile state to approximate a fresh context — strictly more code to reach a strictly worse result. *Reuse `_context` but rebuild only the `CSharpContext`* — still carries stale `CompiledItems`/diagnostics on the `CompileContext`. *Capture and re-pass the original `CompileContext` instance* — that instance is already `Compiled` and state-laden (the same bug); only a *fresh* `CompileContext` built from its options works. *Capture `context.OutputProfile` in the watcher block and seed it onto the fresh context* — **incorrect**: the watcher block runs **after** the first inner compile, so `context.OutputProfile` has already been mutated in place by any top-level `@profile()` directive ([`ProfileExtension.InitStart`](../../../src/Heddle/Extensions/ProfileExtension.cs):51). Seeding that captured value makes a reload of source that *removed* (or changed) the directive render under the **stale** profile, diverging from a from-scratch compile — the exact "honor the original options" break. Re-deriving `OutputProfile` from `Options` (the ctor default) + the freshly-parsed directive is the only from-scratch-equivalent behavior, so nothing is captured. *Capture `context.RootScopeType` in the watcher block (post-compile) and seed it* — **incorrect** for the identical reason: a root `@model` has already reassigned `RootScopeType` in place by then ([`ModelExtension.cs`](../../../src/Heddle/Extensions/ModelExtension.cs):23–24, writing through [`CompileScope.cs`](../../../src/Heddle/Runtime/CompileScope.cs):30 to the root context), so a reload that edits the directive away would seed the stale `@model` type instead of the original ctor model type. The model type is therefore captured **before** the first inner compile, not in the watcher block.
- **Grounding.** [`ContextCompilation.Compile`](../../../src/Heddle/Runtime/ContextCompilation.cs) finalize guards; [`CompileContext`](../../../src/Heddle/Runtime/CompileContext.cs) (ctor `OutputProfile = Options.OutputProfile` ≈146; "never read at render time" ≈201) / [`CompileScope`](../../../src/Heddle/Runtime/CompileScope.cs) state; [`ProfileExtension`](../../../src/Heddle/Extensions/ProfileExtension.cs) (`@profile()` mutates `context.OutputProfile` ≈51); [`HeddleTemplate.Recompile`](../../../src/Heddle/HeddleTemplate.cs) fresh-scope precedent.

### D4 — Recompile trigger: any watcher event resolving to the target existing; delete keeps last-good (closes P1-Q2, P1-Q3)

- **Decision.** Route the wired handlers so the **public event** mapping matches the documented contract and the **recompile trigger** fires whenever the watched file exists after the event:
  - `Changed += FileChanged` → raise `OnFileChanged`; **recompile** (in-place edit).
  - `Created += FileCreated` (**new wiring**) → raise `OnFileChanged` (there is no `OnFileCreated` public event; a (re)appearance of the watched file is a content change); **recompile**. Covers editors that save by delete-then-create.
  - `Renamed += FileRenamed` → raise `OnFileRenamed`; **recompile only when the rename lands ON the target** — `string.Equals(Path.GetFileName(e.FullPath), _watchFileName, StringComparison.OrdinalIgnoreCase)` (rename-onto-target, the atomic-save idiom: write temp, rename over `home.heddle`). A rename **away** from the target (`e.FullPath` is some other name) does not recompile — it is effectively a delete; last-good stays published.
  - `Deleted += FileDeleted` → raise `OnFileDeleted`; **do not recompile** — keep the last-good document renderable (P1-Q2).

  The recompile body is a single private `Reload()` (D3 + D5). A **failed** recompile — edited source that no longer compiles, or a transient read failure — keeps the last-good document published and surfaces the failure on `CompileResult` (see D5). The current `if (e.ChangeType == WatcherChangeTypes.Changed)` guard inside `FileChanged` is dropped: the `Changed` handler only ever receives `Changed` events, so the guard is redundant.

  **Empty-content policy.** `Reload()` recompiles only `if (!string.IsNullOrWhiteSpace(document))`; an empty/whitespace-only read is a **no-op that keeps last-good published** — it neither recompiles nor sets a failure `CompileResult`. This is deliberate, not a gap: an empty save is overwhelmingly a **mid-write editor truncation** (many editors truncate-then-write, briefly raising a `Changed`/`Created` on a zero-byte file before the real content lands and raises a second event), and clobbering a working document with an empty reload — or surfacing a spurious failure — on every save would defeat the reload loop. A *deliberate* blank template has no useful render either, so preserving last-good is the correct outcome in both cases. The subsequent real-content event reloads normally. `CompileResult` continues to report the last actual compile (the same "last-good stays published, no recompile" outcome as the `Deleted` event above); a host that wants to observe the empty save watches `OnFileChanged` (which still fires before the guard). Asserted by `EmptyOrWhitespaceSaveIsNoOpAndKeepsLastGood` ([testing plan](#testing-plan)).
- **Rationale.** A filter-and-arm-only fix would still miss the common editors that save via rename or delete-create (they emit no `Changed`), which R6 forbids (a fix that *looks* broken). Triggering on "the target exists now" (Changed / Created / Renamed-onto-target) is the exact predicate that makes atomic saves work while leaving a genuine delete as a non-recompiling, last-good-preserving event. `OrdinalIgnoreCase` matches Windows/macOS default file-name casing; case-sensitive platforms are covered by the documented platform caveat (D6-note). Mapping `Created` to the public `OnFileChanged` avoids inventing an `OnFileCreated` event (interface/back-compat surface) for a case the existing event already describes ("the file's content changed on disk").
- **Alternatives rejected.** *Recompile on `Deleted`* — the file is gone; reading it throws and there is nothing to compile. Keeping last-good is the useful behavior. *Recompile on every `Renamed` regardless of direction* — a rename-away would read-fail and clobber `CompileResult` with a spurious error, hiding the still-valid last-good state. *Add an `OnFileCreated` event* — widens the public `IHeddleTemplate` interface (a breaking addition to implementors) for no contract the plan requires.
- **Grounding.** [`HeddleTemplate`](../../../src/Heddle/HeddleTemplate.cs) handlers (≈388-424); [FileSystemWatcher events](https://learn.microsoft.com/en-us/dotnet/api/system.io.filesystemwatcher) (Changed/Created/Renamed/Deleted, filter semantics); plan risks table (atomic-save).

### D5 — Safe live swap: publish a coherent old-or-new document; release the superseded one via Phase 4's deferred-dispose discipline (closes P1-Q4)

- **Decision.** Two coordinated pieces built on Phase 4's primitive (consumed, not re-specified) plus **one serialization primitive Phase 1 adds** — `private readonly object _publishGate = new object();`. Publication (capture-superseded → field writes → enqueue → drain) is serialized under `_publishGate`; **renders never take this lock** (they use only `Interlocked _runners` + volatile reads), so it stays entirely off the render hot path. It is needed because `FileSystemWatcher` raises `Changed`/`Created`/`Renamed` on **threadpool threads concurrently** (rapid saves), and the concurrency stress test additionally calls `Recompile` concurrently — so two publishers can race the store block, tearing the field group and double/skip-enqueuing a superseded document if unserialized.

  **(a) Coherent publish (serialized) + coherent read (snapshot after `EnterRender`).** The private `Compile` store block runs under `_publishGate`. It first honors a disposal that has already been requested (a watcher callback can land after `Teardown()` disposed the live document — `_watcher.Dispose()` does not join callbacks already in flight); in that case it disposes the fresh artifact rather than publishing onto a torn-down template (no leak, no resurrection). Otherwise it captures the document being replaced, publishes document-then-strategy, and enqueues the superseded document for the gated drain:

  ```csharp
  if (!simulate)
  {
      lock (_publishGate)                                   // serialize all publishers (watcher callbacks + Recompile + first compile)
      {
          if (Volatile.Read(ref _disposeAfterComplete) != 0)
          {
              // Dispose()/Teardown() already ran (or is committed): do not publish onto a dead template.
              compileScope.Dispose();                       // release the freshly compiled artifact instead of leaking it
              rtdoc?.Dispose();
              stored = true;                                // Phase-4 WI2 finally must NOT dispose again
              return new HeddleCompileResult(true, document, parseContext);
          }
          var superseded = _runtimeDocument;                // the document about to be replaced (null on first compile)
          _context = compileScope;                          // plain ref write; atomic per ECMA
          _document = optimizedDocument;
          _runtimeDocument = rtdoc;                          // volatile publish — document first
          _processStrategy = rtdoc?.Strategy;                // volatile publish — then strategy
          _encoder = compileScope.CompileContext.Options.Encoder;
          _renderBudget = compileScope.CompileContext.Options.RenderBudget;
          _validateModelType = compileScope.CompileContext.Options.ValidateModelType;   // Phase-4 WI3 line
          if (superseded != null)
          {
              _supersededDocs = _supersededDocs ?? new ConcurrentQueue<RuntimeDocument>();
              _supersededDocs.Enqueue(superseded);
          }
          DrainLocked();                                     // dispose queued docs iff no render can hold them (b)
          stored = true;                                     // retained by the template — Phase-4 WI2 finally must not dispose
      }
  }
  ```

  (The `stored` flag is Phase 4 WI2's disposal guard; the two dispose-race exits set it so the `finally` neither leaks nor double-disposes.) `Render` takes its document snapshot **after** `EnterRender()` increments `_runners`, so the drain's `_runners` gate provably covers every render that could hold the snapshot (see proof); it reads the document **once** into a local and derives strategy + `NeedsLocals` + the model-type context from that coherent read:

  ```csharp
  // compile guard — liveness only (was this template ever compiled?); a plain volatile read, not the swap snapshot
  if (_processStrategy == null)
  {
      if (CompileResult == null) throw new TemplateInitException("Compile first");
      throw new TemplateCompileException(CompileResult.ErrorList);
  }
  EnterRender();                                             // Interlocked.Increment(_runners) THEN re-read the dispose flag (Phase 4)
  try
  {
      var doc = _runtimeDocument;                            // snapshot AFTER the increment → drain-safety proof holds
      var ctx = _context;                                   // read once with the snapshot; guard + render share one context
      var strategy = doc != null ? doc.Strategy : _processStrategy;   // precompiled: doc == null, strategy is the field
  #if DEBUG
      var validateModelType = true;                          // Phase-4 WI3: DEBUG always validates
  #else
      var validateModelType = _validateModelType;            // Phase-4 WI3: Release honors the opt-in
  #endif
      if (validateModelType && !_precompiled && data != null && !ctx.ScopeType.Type.IsType(data))
      {
          throw new TemplateProcessingException(string.Format(CultureInfo.InvariantCulture,
              "Type mismatch. Need {0} but got {1}",
              ctx.ScopeType.Type?.FullName ?? ctx.ScopeType.ToString(), data.GetType().FullName));
      }
      var scope = new Scope(data, callerData, data, chained, renderer, null,
          (doc?.NeedsLocals ?? false) ? new ScopeLocals() : null);
      strategy.Render(scope);
  }
  finally { ExitRender(); }
  ```

  The model-type guard now reads `ctx.ScopeType` from the **same** snapshot as the render (a single read of `_context`), rather than re-reading `_context` independently — so the guard and the strategy it protects observe one coherent document/context (Phase 4's guard read `_context.ScopeType` at each use; Phase 1 lifts that read into the snapshot). Moving the guard after `EnterRender` costs nothing: a mismatch still throws before `strategy.Render`, and the `finally` balances the increment.

  **(b) Superseded-document release, serialized under `_publishGate`.** Field: `private volatile ConcurrentQueue<RuntimeDocument> _supersededDocs;` — `null` (zero cost) for every template that never reloads; assigned only under `_publishGate`, read `volatile` by `ExitRender` to decide whether to take the lock at all. The drain disposes queued documents **only** while no render is active, and — crucially — **only ever runs under `_publishGate`**, so the queue cannot grow underneath a drainer:

  ```csharp
  private void DrainLocked()                                 // caller holds _publishGate
  {
      var q = _supersededDocs;
      if (q == null || Volatile.Read(ref _runners) != 0)
          return;                                            // a render may still hold a superseded doc; a later publish / ExitRender / Teardown drains
      while (q.TryDequeue(out var doc))
          doc?.Dispose();
  }
  ```

  `DrainLocked()` runs at three sites, **all under `_publishGate`**: the publish store block (a); Phase 4's `ExitRender()` when the render count reaches zero (guarded by the `volatile` `_supersededDocs != null` check so non-reloading templates never take the lock); and `Teardown()` (whole-template dispose), which drains the remainder. Because every drain holds the lock and every enqueue holds the lock, no drainer ever iterates a queue a concurrent publisher is still appending to — closing the "earlier drainer disposes a later publisher's document" TOCTOU.
- **Rationale (correctness proof).** Two properties, under concurrent publishers + renders + dispose.

  *No use-after-release.* A drain disposes doc `D` only if, at a `Volatile.Read(_runners)` taken **while holding `_publishGate`**, `_runners == 0`. Suppose a render `R` holds `D` (i.e. `R` snapshotted `doc == D` after its `EnterRender` increment) and a drain disposes `D`. `D` is in the queue, so `D` was superseded by some publish `P` (which overwrote `_runtimeDocument` and enqueued `D`) that ran **under `_publishGate` before** the drain acquired the lock. `R` read `_runtimeDocument == D`, so `R`'s snapshot read preceded `P`'s overwrite of `_runtimeDocument`. Order `R` against the drain's `_runners == 0` read: if `R`'s increment happened-before that read, the drain would observe `_runners ≥ 1` and defer — contradiction. So `R`'s increment happened-**after** the drain's `_runners` read; then `R`'s snapshot read (later still, program order after the increment, and the increment is a full fence) happened after the drain's read, which happened after `P` released the lock, which happened after `P` overwrote `_runtimeDocument` past `D`. Therefore `R` would have snapshotted a document **newer than `D`**, not `D` — contradiction. Hence no active render holds any disposed document: **no use-after-release.** This is exactly why the snapshot must be taken **after** `EnterRender` (Blocker-1 ordering): a snapshot taken *before* the increment could read `D`, then a same-instant drain seeing `_runners == 0` could dispose `D` under it.

  *Exactly-once + eventual release.* Every superseded document is enqueued exactly once (enqueue is under the lock; the single-publisher-per-lock discipline prevents the double-enqueue a concurrent store block would otherwise cause). `TryDequeue` removes each item once, so each is disposed at most once. A queued document is disposed by whichever of the three drain sites next observes `_runners == 0` under the lock; if renders are perpetually active it is disposed at latest by `Teardown()` (which also holds the lock and drains the remainder). So each superseded document is disposed **exactly once** and **never while a render holds it**.

  *Coherent read.* The single post-increment volatile read of `_runtimeDocument` yields an entirely-old-or-entirely-new document — the `(NeedsLocals, Strategy)` pair comes from one document — which is *volatile visibility*, not multi-field atomicity (consistent with P1-Q4's "no full multi-field atomicity beyond volatile-field visibility unless a test shows tearing"). `_encoder`/`_renderBudget` are rewritten to **identical** values every recompile (original options honored, D3), so old-or-new is the same value. The model-type guard reads `_context.ScopeType` — **not** `RootScopeType` — and `@model` mutates `ScopeType` ([`CompileContext.cs`](../../../src/Heddle/Runtime/CompileContext.cs):218-229), so an edited file that changes `@model` re-derives a different `ScopeType` on the fresh reload context. Guard-vs-swap coherence is bounded thus: `_context` is a plain reference field written under `_publishGate` and read **once** per render into `ctx` (an atomic reference read, ECMA-335 §I.12.6.6), so `ctx.ScopeType` is self-consistent for that render; for the common reload (content edit, `@model` unchanged) old and new contexts carry the same `ScopeType`, so the guard is coherent regardless of which the render observes; for a reload that **changes** `@model`, a render straddling the swap validates its `data` against either the old or the new `ScopeType` — each a self-consistent old-or-new outcome, never a torn type — and this guard is `DEBUG`/opt-in diagnostic only, so it can never corrupt render state (P1-Q4's accepted old-or-new coherence). Renders on the dynamic path always find `doc != null` after passing the compile guard (a successful compile sets `_runtimeDocument` and `_processStrategy` together under the lock; a failed recompile leaves both untouched), so `doc?.Strategy`/`doc?.NeedsLocals` never see a half-published pair.
- **Alternatives rejected.** *Orphan the old document (today's behavior — overwrite `_runtimeDocument`)* — leaks the document (and its `CompileScope`/assembly-holding context) every reload; the plan's risk table calls this out. *Dispose the superseded document immediately at the swap* — a render that captured it before the swap would use-after-dispose. *Lock-free drain gated by a single `Volatile.Read(_runners) == 0` then `while(TryDequeue) Dispose()` (an earlier draft of D5)* — **unsafe**: a drainer that passed its gate at `_runners == 0` keeps looping while a **later** publisher enqueues a document that a **new** render (incremented after the gate) now holds, and disposes it under that render (the queue grows beneath the un-re-checked loop). Serializing publish+enqueue+drain under `_publishGate` removes the growing-queue window — a drainer only ever sees a queue no concurrent publisher is appending to, and its `_runners` read is the authoritative gate. *Snapshot the document before `EnterRender`* — breaks the drain-safety proof (a render can hold a document a concurrent `_runners == 0` drain disposes). *Block the swap until renders drain (a barrier/`SemaphoreSlim` on the watcher thread)* — a blocking wait on the callback, inconsistent with Phase 4's non-blocking discipline; `_publishGate` never waits on a render (renders don't take it; `DrainLocked` returns immediately when `_runners != 0`). *Full multi-field atomic swap of `(document, strategy, encoder, budget, context)` behind one immutable holder* — explicitly deferred by P1-Q4; the snapshot achieves the needed coherence without it. **Most-reversible escalation if the concurrency test ever shows tearing** the snapshot does not cover: promote the published quintuple into a single immutable holder swapped by one volatile reference write and read once per render. Trigger: the [concurrency test](#testing-plan) exhibits a torn read the document snapshot does not already prevent. This is reversible (it only tightens publish granularity) and is recorded, not pre-built (YAGNI).
- **Grounding.** [Phase 4 D1 / SD-A](phase-4-engine-robustness.md#d1--disposal-synchronization-interlocked-runner-count--cas-guarded-single-teardown-closes-p4-q3-sd-a); [`HeddleTemplate.Render`](../../../src/Heddle/HeddleTemplate.cs) and store block; [`RuntimeDocument.Strategy`/`NeedsLocals`/`Dispose`](../../../src/Heddle/Runtime/RuntimeDocument.cs); [Interlocked](https://learn.microsoft.com/en-us/dotnet/api/system.threading.interlocked) / [Volatile](https://learn.microsoft.com/en-us/dotnet/api/system.threading.volatile) (netstandard2.0).

### D6 — No debounce; documented platform limits (closes P1-Q5; records the FileSystemWatcher caveats)

- **Decision.** No debounce/coalescing this phase. After D3, every recompile converges to the same finalized document from the same on-disk content, so duplicate/rapid `Changed` events cause **redundant but output-idempotent** recompiles (the rendered document is identical; never incorrect); the superseded-document release (D5) keeps redundant reloads from **leaking runtime documents**. One honest caveat: redundant reloads are *output-idempotent, not free* — each `FullCSharp` reload emits and permanently loads a **new** assembly (fresh `ClassGuid` per fresh context ⇒ cache miss; see [Performance](#performance-considerations)), so rapid saves of a FullCSharp template accumulate loaded assemblies until the process exits. This is a cost, not a correctness bug, and is the concrete future trigger for debounce. A second bounded caveat follows from `_publishGate` ordering publishes by **lock-acquisition order, not source recency** (there is no monotonic write-stamp): the "output-idempotent" property assumes concurrent reloads read the *same* on-disk content, which holds for rapid duplicate events of one save. If two *distinct* saves interleave — reload₁ reads v1, then v2 lands and reload₂ reads+publishes v2, then a slower reload₁ acquires the gate last and publishes **v1** — the template serves stale v1 until the next watcher event. This is a dev-time-only, self-correcting window (the next change re-converges), accepted as a bounded limit of a local reload feature. **Revisit trigger:** if observed in practice, add a monotonic source-version stamp captured at read time and a last-writer-wins-by-recency guard in the store block (publish only if the read stamp is newer than the currently-published one). The known `FileSystemWatcher` platform limits — no reliable delivery on **network/UNC drives**, internal-**buffer-overflow** event loss under bursts, and **case-sensitivity** differences across OSes — are documented as scoped limits of a **local development-time** reload feature; the engine does not paper over OS-level watcher behavior.
- **Rationale.** Debounce is a policy the corrected single-file mechanism *enables later* (plan boundary), not a correctness requirement now; adding it would be a speculative knob (YAGNI). The FullCSharp per-reload assembly-load cost is bounded (local dev, one editor, human save cadence) and is the measured need that would later justify debounce or assembly unloading. Naming the platform limits sets honest expectations without pretending to fix OS behavior the engine does not control.
- **Alternatives rejected.** *Add a debounce timer/coalescing window now* — deferred by P1-Q5; introduces timing state and a new option for no correctness gain. *Attempt to support network drives / overflow recovery* — outside a local dev-reload scope; unbounded complexity.
- **Grounding.** [FileSystemWatcher remarks — buffer / network drive / notify limits](https://learn.microsoft.com/en-us/dotnet/api/system.io.filesystemwatcher); plan risks table.

| `_watchFileName`, `_watchOptions`, `_watchModelType`, `_watchControllerName` | — (new) | New fields. `_watchFileName`/`_watchOptions`/`_watchControllerName` are captured in the watcher block; `_watchModelType` is captured **before the first inner compile** — its original pre-`@model` value, since `@model` reassigns `RootScopeType` in place ([`ModelExtension.cs`](../../../src/Heddle/Extensions/ModelExtension.cs):23–24) (D1/D3). `OutputProfile` is **not** among them — re-derived from `_watchOptions` + freshly-parsed `@profile()` each reload (D3), so neither a directive-flipped profile **nor** a directive-flipped model type is ever carried across a reload. |
| store block (`!simulate`) | assigns `_context`/`_document`/`_runtimeDocument`/`_processStrategy`/`_encoder`/`_renderBudget`/`_validateModelType`; disposal guarded by the `stored` flag (Phase-4 WI2 `finally`) | wrap in `lock (_publishGate)`; **first** honor an already-requested dispose (`_disposeAfterComplete != 0` → dispose the fresh artifact, set `stored`, return — no publish onto a torn-down template); else capture `superseded`, publish document-then-strategy, enqueue, `DrainLocked()`, set `stored` (D5a). |
| `Render` body | `_processStrategy == null` guard, then `EnterRender()`; body reads `_processStrategy` for `.Render` and `_runtimeDocument?.NeedsLocals`; model-type guard reads `_context.ScopeType` before `EnterRender` | keep `_processStrategy == null` liveness guard before `EnterRender()`; **after** `EnterRender()` snapshot `doc = _runtimeDocument` and `ctx = _context` once; `strategy = doc?.Strategy ?? _processStrategy`; model-type guard on `ctx.ScopeType`; scope from `doc?.NeedsLocals`; `strategy.Render` (D5a). finally `ExitRender()`. |
| `ExitRender()` | `if (Interlocked.Decrement(ref _runners) == 0 && Volatile.Read(ref _disposeAfterComplete) != 0) Teardown();` | drain superseded before the dispose check, **only** when there is a queue: `if (Interlocked.Decrement(ref _runners) == 0) { if (_supersededDocs != null) lock (_publishGate) { DrainLocked(); } if (Volatile.Read(ref _disposeAfterComplete) != 0) Teardown(); }`. |
| `Teardown()` | one-shot CAS, then disposes `_watcher`, `_runtimeDocument`, `GC.SuppressFinalize` | after the CAS, dispose `_watcher` + `_runtimeDocument` **and** drain-dispose any remaining `_supersededDocs` **under `lock (_publishGate)`**, then `GC.SuppressFinalize`. The lock makes the store block's post-`Teardown` guard and the teardown disposal mutually exclusive (no publish-vs-teardown race), so a late watcher callback cannot leak or resurrect a document. |

No change to `_disposeAfterComplete`/`_teardownDone`/`_runners` atomic-state semantics; Phase 1 only *reads* `_runners` (via `Volatile.Read`) under `_publishGate` in `DrainLocked`, and wraps the (Phase-4-owned) teardown disposal body plus the drain sites in `_publishGate`.

## Public API / contract

**No new or changed public signatures.** `EnableFileChangeCheck`, `OnFileChanged`, `OnFileDeleted`, `OnFileRenamed`, `Recompile`, `Generate`, `Compile`, `TryCompilation` keep their current shapes on [`HeddleTemplate`](../../../src/Heddle/HeddleTemplate.cs) and [`IHeddleTemplate`](../../../src/Heddle/IHeddleTemplate.cs). What changes is **behavior**, restoring the already-documented contract:

- **`EnableFileChangeCheck = true` on a file-compiled template** now installs an **armed** watcher whose filter matches the real file (`TemplateName + FileNamePostfix`). An in-place edit, a create/re-create, or a rename-onto-target recompiles the template into a document **equivalent to a fresh compile of the new content honoring the original options** (model type, `ExpressionMode`, `OutputProfile`, `Encoder`, `RenderBudget`, `TrimDirectiveLines`, `Functions`, `MaxRecursionCount`), and subsequent renders reflect it. This holds for the 1st, 2nd, 3rd, … edit and for `FullCSharp` templates whose embedded C# changed (Roslyn + delayed-extension finalization re-run each reload).
- **`OnFileChanged`** fires on an in-place edit **and** on a create/re-create of the watched file (no `OnFileCreated` event exists); **`OnFileDeleted`** fires on a delete (no recompile — last-good preserved); **`OnFileRenamed`** fires on any rename touching the watched name, and a rename **onto** the target additionally recompiles. **Event `sender`.** All four wired handlers raise their public event with `sender = this` (the `HeddleTemplate`), consistent with the existing `OnFileDeleted`/`OnFileRenamed` wiring (`HeddleTemplate.cs:396` already passes `this`). This harmonizes `OnFileChanged`, whose current dead-code handler forwards the raw `FileSystemWatcher` as sender (`:408`); since the watcher was inert (never armed), no subscriber ever observed a sender, so this is a first-observable contract, not a change to any live behavior — but it is stated here because a `FileSystemEventHandler`'s sender conventionally is the watcher, and Heddle deliberately exposes the template instead (the watcher is never a public handle).
- **A failed recompile** (broken edited source, or a transient read failure) keeps the last successfully-compiled document renderable and reports the error on `CompileResult` (`Success == false`, `Errors`/`ErrorList` populated) — it never corrupts state or throws to an unhandled watcher-thread callback.
- **`Dispose()` / concurrent renders during a reload:** a recompile that swaps the live document publishes a coherent old-or-new document to in-flight renders and releases the superseded document only once no render can hold it (Phase 4's deferred-dispose discipline, extended with `_publishGate` serialization for concurrent publishers). As a beneficial side effect, the public `Recompile` overloads — which route through the same store block — also stop **orphaning** the previous document: the superseded document is released through the same gated drain, so a `Recompile` under an active render is leak-free and swap-safe. (This side benefit is the leak-free release only; `Recompile(newDocument, context)` and `Recompile(ExType)` still compile under the **passed/default** options, not the watcher's captured original options — only the watcher path recompiles from-scratch-equivalent under the original options, D3.)
- **Thread-safety.** No mutable per-render state is added. `_watchOptions`/`_watchModelType`/`_watchFileName`/`_watchControllerName` are written once at compile and read only on the watcher-callback thread. Publication (capture-superseded → `_context`/`_document`/`_runtimeDocument`/`_processStrategy`/`_encoder`/`_renderBudget`/`_validateModelType` writes → enqueue → drain) is serialized under `_publishGate`, so concurrent watcher callbacks and a concurrent `Recompile` cannot tear the *rendered* field group or double/skip a superseded enqueue. **One field is deliberately outside this group:** `CompileResult` is assigned by `Reload()` *after* `Compile` returns (and by `Compile` itself), so it is **last-writer-wins diagnostic metadata**, not part of the serialized render state — under concurrent publishers the visible `CompileResult` can transiently describe a *different* compile than the currently-published document (e.g. a racing failed reload's error report shadowing a just-published success). This is a bounded, accepted limit: the `(document, strategy, NeedsLocals)` pair a render observes is always coherent (that is what `_publishGate` + the snapshot guarantee); only the diagnostic result may momentarily lag or lead the published document. Bringing `CompileResult` under `_publishGate` was rejected as it would put a diagnostic write on the swap path for no render-correctness gain. Renders never take `_publishGate`: they publish nothing and read `_runtimeDocument`/`_processStrategy` once per render (volatile snapshot taken **after** `EnterRender` increments `_runners`). `_supersededDocs` is drained only under `_publishGate` and only when `Volatile.Read(_runners) == 0`, so a superseded document is disposed exactly once and never while a render holds it. `TemplateOptions` is caller-owned config, reused by reference across recompiles and never mutated by the engine (child compiles copy it) — safe to share.

## Back-compat and migration

Governed by [R5](../../plan/common/standing-rulings.md#standing-rulings-user-set) (additive / pure-correctness; no rendered-byte change to any valid template; no flipped default) and [R6](../../plan/common/standing-rulings.md#standing-rulings-user-set) (the three defects ship as one deliverable). **No breaking-window entry** — nothing here is a breaking change, so [breaking-windows.md](../common/breaking-windows.md) is untouched.

- **Opt-in only.** Every behavior change is gated behind `EnableFileChangeCheck = true`, which [defaults to `false`](../../csharp-api.md#templateoptions). A template that does not set it takes the identical code path as today: the `if (EnableFileChangeCheck)` block is skipped, no watcher/`_supersededDocs` is allocated, and the `Render` snapshot reads the same fields to the same effect. The only new per-instance cost for **any** template is the single `_publishGate` object initialized at construction (needed so a concurrent public `Recompile` cannot tear the field group) — one object per `HeddleTemplate`, off the render path and dwarfed by the compiled document; it adds **zero** render-path allocation (the benchmark-guarded invariant). **Proven by:** `FlagOffAndInlineOutputsAreByteIdentical` + the full golden suite staying byte-identical + the render benchmarks (allocated bytes unchanged).
- **No rendered-byte change.** Inline compiles, precompiled templates, and flag-off file compiles render byte-identically. The only observable change for a host that *already set the flag* is that reload now works (it silently did not) — the restoration of the documented contract, not a byte change to any source's output. **Proven by:** byte-identical goldens + the deterministic reload tests asserting the new content matches a from-scratch compile of that content (never a *changed* rendering of unchanged content).
- **Public `Recompile` stops leaking the superseded document.** The shared store block now releases the superseded document (via the `_publishGate`-gated drain) instead of orphaning it, so the public `Recompile` overloads stop leaking the prior document under an active render — a strict improvement with no signature or success-path behavior change. This is bounded to the leak-free/coherent-swap guarantee: `Recompile` still uses its passed/default options (not the watcher's captured originals), so it is not claimed to be a from-scratch-equivalent-under-original-options reload (that is the watcher path, D3). **Proven by:** `RecompileSwapInterleavedWithConcurrentRendersIsCoherentAndLeakFree`.
- **Restored contract, not a new one.** `OnFile*` / `EnableFileChangeCheck` were already public and documented to work; this makes them behave as documented. The `csharp-api.md` "Current limitation" and `language-assessment.md` §4.6/§8 "inoperative watcher" notes are reconciled in the same change (WI4).

## Performance considerations

- **Render hot path (touched):** the `Render` snapshot replaces the previous three volatile reads of the strategy/document (`_processStrategy` at the guard, `_runtimeDocument` for `NeedsLocals`, `_processStrategy` again for `.Render`) with **one** volatile read of `_runtimeDocument` plus non-volatile field accesses (`doc.Strategy`, `doc.NeedsLocals`) — a net **reduction** in volatile reads on the dynamic path and neutral on the precompiled path. **Zero new allocation** (no lock, no wait handle; the `Scope`/`_processStrategy.Render` core is unchanged). The swap adds nothing to the render path: publication happens off-thread on the watcher callback.
- **`ExitRender` (touched):** one extra `volatile` reference-field read (`_supersededDocs`) inside the existing `_runners == 0` branch; for the overwhelmingly common non-reloading template `_supersededDocs` is `null`, so the branch is a single compare-and-skip — **no lock is taken** (`_publishGate` is entered only when a superseded document is actually pending). `DrainLocked` does real work only when a superseded document is pending **and** `Volatile.Read(_runners) == 0`.
- **Watcher callback (off hot path):** reading the file and compiling into a fresh context is a compile-path cost ("compile once, render many"), incurred only on an actual on-disk change; not a render cost, and the `_publishGate` lock is uncontended in the common single-editor case. A `FullCSharp` reload pays Roslyn parse/emit **and permanently loads a new assembly**: the process-global `ContextCompilation.Cache` is keyed by generated code text, which embeds a per-`CSharpContext` `ClassGuid` (`Guid.NewGuid()`), and D3 mandates a fresh context (hence a fresh `ClassGuid`) per reload — so each `FullCSharp` reload is a cache **miss** that re-emits and `Assembly.Load`s a distinct, never-unloaded assembly. This is a documented cost/limit of local-dev hot reload, not a leak the engine can avoid without assembly unloading (a deferred `AssemblyLoadContext` concern); it does not affect render-path performance and is bounded to the FullCSharp tier (Native reloads pay no Roslyn cost).
- **Guarding benchmarks:** `TextRenderBenchmarks.RenderHeddle` and `SinkRenderBenchmarks.*` (all route through `Render`); acceptance per the [regression gate](#testing-plan): allocated bytes unchanged, mean time within reported error. Coordinated with Phase 4's identical benchmark guard on `Render`.

## Standards compliance

- **YAGNI / simplicity (precedence rule 3).** The fix converges on the engine's existing from-scratch compile (fresh context) and Phase 4's existing disposal primitive rather than building a reload framework, a debounce timer, or a file-provider seam — all explicitly deferred. The superseded release is the smallest *correct* addition: a queue drained under Phase 4's `_runners` gate, plus **one** publisher-serialization lock (`_publishGate`) — the minimal mechanism that survives concurrent watcher callbacks + `Recompile`. A lock-free drain was drafted and rejected because it is provably unsafe under concurrent publishers (an earlier drainer disposes a later publisher's document under a new render); the single lock, kept entirely off the render hot path, is the smallest thing that is actually correct (precedence rule 1 over a smaller-but-racy diff).
- **DRY at the load-bearing seam.** The watcher filter is derived from the *same* `_reader.GetFileName()` the read path uses — one representation of "which file this template is," which is the exact knowledge whose duplication caused the filter bug ([coding standards](../common/coding-standards.md#dry-applied)).
- **Correctness beats a smaller diff (precedence rule 1).** D3 chooses compile-into-a-fresh-context over the superficially smaller "reset two flags," because the latter is *incorrect* against the accumulated-scope-state evidence — a smaller diff that reintroduces a subtler bug is not simpler.
- **Measured hot-path performance (precedence rule 4).** The `Render` snapshot is justified by the coherence requirement and independently reduces volatile reads; the benchmarks guard it. No lock is taken on the render path — `_publishGate` is held only by publishers (watcher callback / `Recompile`) and in `ExitRender`/`Teardown` solely when a superseded document is actually pending (never for a non-reloading template), consistent with Phase 4's lock-free render discipline.
- **Thread-safety contract stated.** No mutable per-render state is added; swap state is published under `_publishGate` and read once per render via a post-`EnterRender` volatile snapshot; superseded release is serialized under `_publishGate` and gated by the atomic `_runners`. The cross-phase coupling (`_publishGate`/`_supersededDocs` added; `ExitRender`/`Teardown`/store block amended) is enumerated in [Field & member reconciliation](#field--member-reconciliation-with-phase-4).

| Debounce / coalescing of rapid `Changed` bursts | A measured need (redundant recompiles becoming a cost) — P1-Q5 defers it; recompiles are idempotent after D3. |
| Async recompile / a general hot-reload or file-provider abstraction | A product decision to add a reload framework; a plan non-goal. |
| Full multi-field atomic swap behind a single immutable holder | The [swap concurrency test](#testing-plan) exhibits a torn read the document snapshot does not prevent (D5 most-reversible escalation). |
| Network-drive / buffer-overflow-resilient watching; cross-OS case handling | A need beyond local development-time reload; D6 documents these as scoped `FileSystemWatcher` platform limits. |
| Precompiled-backend live reload | Out of scope — the precompiled backend has no watcher and is not a live-reload target. |
