# The 2.0 window — consolidated migration specification

The executable consolidation of [D2](cross-cutting-decisions.md#d2--the-single-20-breaking-window).
**v2.0 is all nine roadmap phases combined** — implemented sequentially with
compatibility defaults, shipped as one release, with amendments discovered during
implementation absorbed through the
[ledger](cross-cutting-decisions.md#cross-spec-amendments-ledger). Because everything
the phases add is additive, the *migration* surface of that release reduces to the five
breaking deltas below — this document owns them in one place, with the execution order,
the user-facing migration recipe, and the verification gate. Each delta is *specified*
in its owning phase spec (linked); this document adds only what no single phase owns —
the combined picture and the decisions about the window itself. It follows the
[spec conventions](spec-conventions.md); phase specs remain authoritative for their own
mechanics.

## Window contents

| # | Change | Owner spec | 1.x state | 2.0 state | User action to keep 1.x behavior |
| --- | --- | --- | --- | --- | --- |
| 1 | `OutputProfile` default flip | [Phase 2 D11](../phase-2-safe-output/README.md#design-decisions) | `Text` | `Html` — the unnamed `@(...)` encodes by default | `options.OutputProfile = OutputProfile.Text`, or mark trusted outputs `@raw` |
| 2 | Encoder swap + pluggability | [Phase 2 D11](../phase-2-safe-output/README.md#design-decisions), executed also through [phase 8 D9](../phase-8-streaming-async/README.md#design-decisions)'s span bridge | `WebUtility.HtmlEncode` at the two pinned call sites; string-based span bridge | `HtmlEncoder.Create(UnicodeRanges.All)` default; new `TemplateOptions.Encoder`; `HtmlEncodedRenderer` slots in `TextEncoder.Encode`/`EncodeUtf8` | None for semantics (both encode safely); byte-exact 1.x output requires pinning a custom encoder reproducing `WebUtility` behavior — the deltas are exactly [phase 2's E01–E14 rows](../phase-2-safe-output/README.md#testing-plan) |
| 3 | `TrimDirectiveLines` default flip | [Phase 4 D9](../phase-4-ergonomics/README.md#design-decisions) | `false` (opt-in) | `true` — whole-line directives swallow their line | `options.TrimDirectiveLines = false` |
| 4 | `[Obsolete]` on `AllowCSharp` | [Phase 1 D6](../phase-1-native-expressions/README.md#design-decisions) (timing pinned here, see W1 below) | Un-attributed bridge over `ExpressionMode` | `[Obsolete("Use ExpressionMode.")]`; behavior unchanged | Switch reads/writes to `ExpressionMode` (mechanical; the bridge semantics are phase 1 D6's) |

## Window decisions

### W1 — `AllowCSharp` obsoletion lands in 2.0

**Decision.** The `[Obsolete]` attribution of `TemplateOptions.AllowCSharp` (phase 1 D6:
"no earlier than the 2.0 window") is pinned **to** the 2.0 window.
**Rationale.** It is warning-only, so it needs no breaking window of its own — but D2's
one-migration principle says users should touch their options code once, and the 2.0
migration note already walks `FullCSharp` users through `ExpressionMode`. Attributing in
the same release folds the churn into that single touch. Phase 1's "no earlier than"
constraint is honored. **Alternatives rejected:** a later minor (spreads the churn over
two upgrades for zero gain); skipping obsoletion (leaves two representations of one
setting documented forever, contradicting phase 1 D6's bridge-then-retire intent).

### W2 — Precompiled assemblies from 1.x fall back under a 2.0 engine

**Decision.** No compatibility shim is added for manifests generated by a 1.x
`Heddle.Generator` running against a 2.0 engine: phase 7's layer-1 gate (engine
same-major-and-≤, [identity-and-metadata.md](../phase-7-build-time-compilation/identity-and-metadata.md))
rejects them by design, and the outcome follows the host's
`PrecompiledMismatchPolicy` — `Fallback` recompiles with the HED7101 warning, `Strict`
throws. The 2.0 migration note must state this explicitly: **rebuild precompiled
projects with the 2.0 generator when upgrading the engine**.
**Rationale.** The gate exists precisely so a major-version semantic change (this
window changes rendered bytes — items 1–3) can never be masked by stale generated code;
a shim would re-open that hole. **Alternatives rejected:** accepting 1.x manifests whose
fingerprints happen to match (the fingerprint does not capture encoder identity — item 2
changes bytes with identical options); a manifest-format major bump (unnecessary — the
engine-version gate already covers it; phase 8's schemaVersion 2 is orthogonal and
additive).

### W3 — Execution order inside the release

**Decision.** One PR series, gated as one combined run, in this order: (1) flip the two
`TemplateOptions` ctor defaults (`OutputProfile`, `TrimDirectiveLines`) — each a
two-line greppable change per phases 2/4; (2) execute the encoder swap at the two
`WebUtility` call sites + the phase 8 span bridge, and add `TemplateOptions.Encoder`;
(3) attribute `AllowCSharp` (W1); (4) flip the generator `.props` defaults (item 5);
(5) **one** golden re-ratification commit, reviewed against the pre-measured churn —
[E01–E14](../phase-2-safe-output/README.md#testing-plan) (encoder deltas),
phase 2's M/X expected-byte flips under the new default, and
[phase 4's golden pair](../phase-4-ergonomics/README.md#testing-plan) (trim churn);
(6) docs: the single migration note (below) plus the per-phase page updates each phase
spec already assigns to the flip.
**Rationale.** Goldens change once, attributably: steps 1–4 are byte-inert until the
suite runs against them together, and the testing standards' fix-forward rule requires
golden changes to arrive as reviewed, spec-backed diffs — which the pre-measured tables
make mechanical. **Alternatives rejected:** per-change golden updates (three churn
reviews of the same files); flipping defaults last (would make steps 2–4 unverifiable
against the final byte expectations until the very end anyway).

## The migration note (deliverable)

One page, shipped with the 2.0 release notes (folded into the release entry per the
changelog convention — no "Unreleased" bucket), containing exactly: the five-row table
above condensed to "what changed / how to keep old behavior"; the W2 rebuild
requirement for precompiled projects; the pointer to `@raw` and `OutputProfile.Text`
for trusted-output hosts; and the `ExpressionMode` mapping for `AllowCSharp` users.
Nothing else is breaking, so nothing else appears.

## Verification gate

The standard [regression gate](testing-standards.md#regression-gates) plus, specific to
this window: the phase 9 gallery sample
[`samples/html-safe-output`](../phase-9-demo-and-integration/gallery.md) runs under
**both** defaults (its 1.x rehearsal asserts the flip's before/after pair); the phase 2
`OutputProfileEncodingTests` E-rows are re-pinned to the `HtmlEncoder` expectations in
the same commit as the swap; and the phase 7 differential gate (precompiled ==
dynamic twin) runs against the 2.0 defaults to prove generated code and engine flipped
together (item 5).
