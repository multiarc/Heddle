# Phase 7 — generated-code shape reference

Supplementary document of the
[phase 7 spec](README.md); normative for the emitter (WI4–WI6) and the snapshot goldens
(D19). It pins the emission conventions, the piece table with the `"…"u8` tier
([D15](README.md#d15--u8-piece-tier-opt-in-in-v1-validated-always-lone-surrogates-downgrade--corrects-one-roadmap-nuance)),
the `#line` mapping rules, and seven worked template → C# examples. The metadata the
same generator emits (manifest, attribute) is specified in
[identity-and-metadata.md](identity-and-metadata.md).

Everything here is gated by the differential harness
([D20](README.md#d20--differential-authority-and-the-drift-gate)): the shapes below are
normative for *structure*; where a detail is deliberately left to the implementer (local
naming, statement ordering inside a body), byte-identical output against the runtime
backend is the acceptance test.

## Emission conventions

- **One generated file per template**, hint name `{SanitizedName}.g.cs`
  ([D11](README.md#d11--typed-entry-points-are-the-recommended-host-api-naming-pinned--ratified)
  naming), plus one `__HeddleManifest.g.cs` per compilation.
- Every file starts with `// <auto-generated/>` (suppresses style analyzers) followed by
  `#pragma warning disable` for the pragmas the emitter knows it triggers (unused
  parameter on `Execute`-only paths), then the namespace block. **Block-scoped
  namespaces** — generated code obeys the
  [repository style](../common/coding-standards.md#repository-style-inferred-preserved)
  where it costs nothing, and block scoping is also the C# 7.3 baseline form.
- All type references are `global::`-qualified — generated code must be immune to
  `using` collisions in the consuming project.
- **Determinism:** pieces numbered in document order (`P0`, `P1`, …) — document order is
  the `.heddle` source order, so a definition body's pieces number before a later call
  site's pieces (see examples 4–5); extension fields in call order (`E0`, `E1`, …), body
  classes in compile order (`Body0` = document root), manifest entries ordered ordinally
  by key. Two identical inputs produce byte-identical outputs (the snapshot goldens
  enforce this).
- **Language tiers** (from `CSharpParseOptions.LanguageVersion`, per
  [D5](README.md#d5--generated-code-shape-strategy-classes--bound-extension-instances--one-piece-hook)):
  baseline C# 7.3; `#line` span form at ≥ 10 (classic `#line n "file"` below);
  `file`-scoped scaffolding at ≥ 11 (below: `internal` classes named
  `__Heddle_{SanitizedName}_{member}`); u8 pieces at ≥ 11 (and only when
  `HeddleEmitUtf8Pieces=true`); collection expressions at ≥ 12 for piece/argument arrays
  (plain `new[] { … }` below). No tier is load-bearing.
- **Scaffolding visibility:** the entry class is `public static`; everything else
  (strategy classes, piece holders) is nested `private`, `file`-scoped, or
  `internal`-mangled — user code binds only the typed entry points and (via the
  manifest) `IProcessStrategy`. The worked examples use nested `private sealed` body
  classes — the collision-free form available at every language tier.

## The render protocol contract

Generated code reproduces the runtime dispatch **literally** — the differential harness
is a semantics gate, not a reimplementation audit:

1. **Call sites follow `TemplateItem`** (verified in
   [TemplateItem.cs](../../../src/Heddle/Runtime/TemplateItem.cs)): evaluate the
   parameter to `object`, swap the model with `scope.Model(value)` (the pre-swap model
   becomes `ParentModelData`), invoke `extension.RenderData(modelScope)` — or
   `ProcessData` when a parent consumes the value. **An empty parameter list is the
   model pass-through**, verified in
   [EmptyParameter.cs](../../../src/Heddle/Runtime/Parameters/EmptyParameter.cs)
   (`GetParameter` returns `scope.ModelData`): `@else()`, `@out()`, and every other
   empty-parens call site emits `extension.RenderData(scope.Model(scope.ModelData))` —
   never a bare `extension.RenderData(scope)`, which would leave a stale
   `ParentModelData` behind and break every body the extension executes under
   `scope.Parent()`.
2. **Chains follow `TemplateChain`:** for `@a(X):b(Y)`, each non-terminal item's
   `ProcessData` result feeds the next item through the combined transform
   `scope.Model(paramValue, previousResult)` (model = the item's own parameter, chained =
   the previous result); only the terminal item renders.
3. **Extension instances are pre-constructed and shared** — `static readonly` fields,
   created once by the CLR type initializer (thread-safe by construction), never mutated
   after `PrecompiledRuntime.Bind` returns. This is the existing extension thread-safety
   contract: no per-render state on instances.
4. **Bodies are generated `IProcessStrategy` classes** installed via
   `PrecompiledRuntime.Bind(extension, body, renderType, needsLocals, line, column)`,
   which (inside `Heddle.dll`, with internal access) performs what `InitStart` does today:
   stores the body strategy, applies `SetUpRenderType`, sets `Position`, and — when
   `needsLocals` — wraps the body in the engine's locals-provisioning decorator so
   `ScopeLocals` frames are created exactly where the runtime backend creates them
   (phase 3 semantics stay in the engine, in one place). The **same decorator** is
   exposed as `PrecompiledRuntime.WithLocalsFrame(body)` for the one body `Bind` never
   sees — the document root: a root body whose document hosts branch participants is
   wrapped at its `Root` field initializer (example 6), mirroring
   `HeddleTemplate.Generate`'s `_runtimeDocument.NeedsLocals` root-frame provisioning
   without a per-render allocation for templates that need none (phase 3 D15's
   allocation budget). The isolation half of phase 3's rule (a frameless body never sees
   its parent's frame) needs no generated-code counterpart: generated bodies execute
   through `GetInnerResult`/`RenderInnerResult`, where the engine already applies it.
5. **`Execute` mirrors `NormalStrategy.Execute`** (verified in
   [RuntimeDocument.cs](../../../src/Heddle/Runtime/RuntimeDocument.cs)): pieces
   contribute their string, calls contribute `extension.ProcessData(modelScope) as string
   ?? string.Empty`, concatenated in order (`string.Concat` over the parts — allocation
   pattern may differ from `ExStringBuilder.Concat`, output bytes may not). `Render` is
   the hot path; `Execute` exists because extension parents consume it
   (`GetInnerResult`).
6. **Member paths reproduce the member tier hop by hop** (verified in
   [ModelParameter.cs](../../../src/Heddle/Runtime/Parameters/ModelParameter.cs) —
   `GetPropertyChainAccessor` emits `Expression.Condition(receiver == null,
   Expression.Default(hopType), MakeMemberAccess)` per reference-typed receiver): a hop
   off a `null` receiver yields `default(T)` of the hop's property type. The C#
   equivalents: reference-typed hops whose property type is also a reference type use
   `?.` (`m?.Manufacturer?.Name`); a hop whose property type is a **non-nullable value
   type** emits the explicit conditional (`m == null ? 0 : m.Count` — `m?.Count` would
   produce `int?`/boxed-null where the runtime produces boxed `0`); value-typed
   receivers access directly (no null test — matching the runtime's
   `MakeMemberAccess` on value types). The root access is a hard cast
   (`var m = (Shop.Product)scope.ModelData;`), matching the runtime's
   `Expression.Convert`; boxing happens once at the `object` boundary of
   `scope.Model(…)`. **Never a whole-expression guard**: `m != null && …` around a
   compound condition is *not* equivalent — see the null-safety rule box in
   [example 2](#example-2--native-expressions-phase-1-ast--c-11).

### The body model-typing rule

The emitter names a model type **only where the template declares one** — milestone 1
is syntactic (D3), so no reflection and no `ISymbol` walk exists to discover types:

| Body | Static model type | Why |
| --- | --- | --- |
| Document root | The `@model` type; `object` handled as the dynamic tier when no `@model` is declared | The one declared root type (examples 1–3 vs 4–5). |
| Branch body (`@if`/`@ifnot`/`@elif`/`@else`) | The **enclosing body's** type | Branch extensions execute bodies under `scope.Parent()` — "the condition value never becomes the body's model" (phase 3, verified in [IfExtension.cs](../../../src/Heddle/Extensions/IfExtension.cs)); the pre-swap model arrives back in `ModelData`. |
| Definition body (`@%<x(…)>{{…}} :: T %@`) | The declared `:: T`; the dynamic tier for `:: dynamic` | `DefinitionBaseExtension` keeps the invocation parameter as the body's model (`scope.Chain(chained)` — verified in [DefinitionBaseExtension.cs](../../../src/Heddle/Core/DefinitionBaseExtension.cs)), and the parameter is declared to be a `T`. |
| Caller content (`@x(P){{…}}`) | The invoked definition's `:: T` | Executes with `ModelData` = the invocation parameter (same verification). |
| List/iteration item bodies (`@list`, `@for`) and any body whose model type only reflection could discover | **The dynamic tier** (C# `dynamic` member access, example 4's shape) | The runtime backend types these by reflection over the live collection type (`ListExtension.InitStart` → `TryGetElementType`) — unavailable in-generator by construction (D3). The C# runtime binder resolves the same members against the same runtime types, so rendered bytes are identical (differential-gated); only dispatch cost differs, recorded under the README's performance section. Milestone 2 *reports* `HED7008` against the symbol-resolved element type but does **not** retype the emission — D3's "same emitted code, better errors" rule. |

Dynamic-tier hops are per-hop null-safe exactly as the runtime's
[DynamicParameter.cs](../../../src/Heddle/Runtime/Parameters/DynamicParameter.cs) emits
them (`input == null ? null : Dynamic(GetMember(name), input)`): the emitted form is
`m == null ? (object)null : (object)(((dynamic)m).User?.Name)` — the leading conditional
guards the first hop, `?.` carries the rest (dynamic `?.` yields `null` for a `null`
receiver, the same value the runtime tier produces).

## The generated-support API

The public contract of the entry points generated code calls (part of the
[README's public API contract](README.md#public-api-contract); namespace
`Heddle.Precompiled`, assembly `Heddle`):

```csharp
/// <summary>Support entry points called by generated code. Not intended for hand-written use.</summary>
public static class PrecompiledRuntime
{
    /// <summary>InitStart-equivalent: installs a generated body on a pre-constructed extension.
    /// needsLocals wraps the body in the engine's ScopeLocals-provisioning decorator (phase 3).
    /// Called only from generated static initializers (thread-safe via CLR type-init); the
    /// extension is never mutated after Bind returns.</summary>
    public static TExtension Bind<TExtension>(TExtension extension, IProcessStrategy body,
        RenderType renderType, bool needsLocals, int line, int column) where TExtension : AbstractExtension;

    /// <summary>Bind for definition call sites. Constructs the definition carrier itself —
    /// DefinitionBaseExtension is internal to Heddle, so generated code never names it (the
    /// same public-entry/internal-access posture as Bind); returns it as AbstractExtension.
    /// Installs props on the definition-body scope (frozen prototype shared when all-constant;
    /// dynamicSetters run against the caller view per phase 5 D8), threads caller content
    /// through the existing funnel, and assigns the carrier's recursion limit (README D23):
    /// the carrier's guard field is otherwise set only by the InitStart this call bypasses,
    /// and unset means throw-on-first-invocation. maxRecursionCount is the build's
    /// HeddleMaxRecursionCount (positive; default 100 = the TemplateOptions ctor default),
    /// baked as a literal at every generated call site — build wins over runtime options.</summary>
    public static AbstractExtension BindDefinition(IProcessStrategy body,
        IProcessStrategy callerContent, object[] props, PrecompiledPropSetter[] dynamicSetters,
        RenderType renderType, bool needsLocals, int maxRecursionCount, int line, int column);

    /// <summary>The engine's ScopeLocals-provisioning decorator (phase 3), for the one body
    /// Bind never sees: a document root hosting branch participants. Wraps body so every
    /// Render/Execute runs under a fresh frame — mirroring HeddleTemplate.Generate's
    /// NeedsLocals root provisioning; roots without participants stay unwrapped
    /// (zero per-render frame allocation, phase 3 D15).</summary>
    public static IProcessStrategy WithLocalsFrame(IProcessStrategy body);

    /// <summary>Indexed prop read for generated definition bodies (PropsData is internal).</summary>
    public static object Prop(in Scope scope, int index);      // AggressiveInlining; pure read

    /// <summary>Root-model read for @root-anchored paths (Scope.RootData is internal).</summary>
    public static object RootModel(in Scope scope);            // AggressiveInlining; pure read

    /// <summary>Root entry: renderer creation, the internal root Scope ctor, and the
    /// size-adaptive buffer high-water mark, mirroring HeddleTemplate.Generate. Stateless
    /// per call apart from the per-template high-water mark the engine already keeps.</summary>
    public static string GenerateString(IProcessStrategy root, object model, object chained, object callerData);

    /// <summary>Registry-then-dynamic-compile partial resolution (README D7). Thread-safe;
    /// generated call sites memoize the result via LazyInitializer (at-most-once).</summary>
    public static IProcessStrategy ResolvePartial(string key, TemplateOptions options);
}

/// <summary>Dynamic prop-argument evaluator, invoked against the caller view (scope.Parent(),
/// phase 5 D8 — applied by the engine so root/chained context stays available).</summary>
public delegate object PrecompiledPropEvaluator(in Scope callerView);

/// <summary>One dynamic prop assignment: the prop's layout index plus its evaluator.
/// Immutable; instances live in generated static readonly arrays.</summary>
public readonly struct PrecompiledPropSetter
{
    public PrecompiledPropSetter(int index, PrecompiledPropEvaluator evaluate);
    public int Index { get; }

```

Two members exist because a field generated code needs is engine-internal — the same
reason `Bind` exists at all: `Prop` reads `Scope.PropsData` (internal, phase 5) and
`RootModel` reads `Scope.RootData` (internal since the beginning; the runtime's
`RootModelParameter` reads it directly, generated `@root` paths cannot). Both are pure,
allocation-free reads. **Correction recorded during the audit:** an earlier draft of this
contract declared `BindDefinition<TExtension>(TExtension extension, …)` — impossible to
call from a consumer assembly, because the definition carrier
([DefinitionBaseExtension](../../../src/Heddle/Core/DefinitionBaseExtension.cs)) is
`internal` and phase 5 deliberately keeps it so; the engine constructs it inside
`BindDefinition` instead. `WithLocalsFrame` was restored for the participant-hosting
document root, which the earlier draft asserted ("GenerateString provisions the frame")
without an API that could carry the fact — `GenerateString`'s signature (pinned, and
bound by phase 8's D7) has no locals parameter, so the frame rides the strategy, not the
entry point.

## The piece table and the u8 tier (D15 shape)

Every static piece is emitted by the single `EmitPiece` hook:

```csharp
// always:
internal const string P3 = "  <p>Made by ";
// when HeddleEmitUtf8Pieces=true and LangVersion >= 11 and no unpaired surrogate (HED7005):
internal static global::System.ReadOnlySpan<byte> P3U8 => "  <p>Made by "u8;
```

The u8 form is a `ReadOnlySpan<byte>` over UTF-8 bytes the **C# compiler** embeds in the
assembly's static data — no allocation, no encoding call, no static constructor at run
time. u8 literals are runtime constants (not `const`, not interpolatable) — irrelevant
here, they are only ever written to a sink. Because both forms come from one hook, the
same generated type serves the v1 string path and the future byte path without
transcoding. Non-normative phase 8 sketch (what the phase 8 emitter update adds at this
hook — **not** emitted in v1):

```csharp
// phase 8, schemaVersion 2 — sketch only:
public static void Render(global::Shop.Product model, global::System.Buffers.IBufferWriter<byte> sink)
{
    …
    sink.Write(P3U8);   // straight copy out of the assembly's read-only data
    …
}
```

Escaping: pieces are emitted as verbatim-or-escaped string literals chosen by content
(quotes/backslashes/control characters escaped; the emitter never trusts raw
interpolation). The u8 twin uses the identical escaped text with the `u8` suffix, so both
encode the same code points; the parity test compares the compiled bytes against
`Encoding.UTF8.GetBytes(P3)`.

## `#line` mapping rules

- Every emitted statement that corresponds to template text carries a span directive
  (LangVersion ≥ 10):
  `#line (l1, c1) - (l2, c2) charOffset "views/product.heddle"`. Operand semantics
  (verified against the
  [C# 10 enhanced `#line` proposal](https://github.com/dotnet/csharplang/blob/main/proposals/csharp-10.0/enhanced-line-directives.md)):
  lines and characters are **1-based** UTF-16 positions in the `.heddle` file; `(l1,c1)`
  is the first mapped character, `(l2,c2)` the span end (one past the last mapped
  character — the interval convention of the `LinePositionSpan` the directive feeds);
  `charOffset` is the count of UTF-16 characters on the **following generated line**
  that precede the mapped code (its indentation plus any emitted prefix; defaults to 0
  when omitted). The worked examples map each call statement to its whole template call
  block (`@` through the closing `)` of the parameter list) with `charOffset` = the
  statement's indentation; finer expression-level spans are an implementer refinement
  the goldens then freeze. Below C# 10: `#line l1 "views/product.heddle"` (line-level
  fidelity only — recorded, acceptable down-level degradation).
- **Both `Render` and `Execute` map.** `Execute` binds each call result to a local so
  every mapped expression sits in its own directive-carrying statement (a member typo
  must never surface at an unmapped `.g.cs` position through the process path —
  `#line hidden` hides sequence points from the debugger but does *not* remap
  diagnostics).
- Scaffolding (piece writes, casts, `Bind` initializers, class/method syntax) sits under
  `#line hidden` so error remapping and debugger stepping skip it and land on template
  lines; the file ends with `#line default`.
- Milestone 1 acceptance: a member typo in a template produces a C# error (e.g. CS1061)
  whose *reported* file/line/column is the template span — asserted by
  `GeneratorDiagnosticTests` fixtures. Milestone 2 pre-empts these with
  `HED7008` before the C# compiler runs; the `#line` mapping stays (verbatim-C# errors
  and debugging still ride it).
- Stepping granularity follows emitted statements — progressive enhancement, correctness
  first (the roadmap's fidelity caveat, kept).

## Scope of the worked examples

Every worked example below is a **complete compilation unit** — every member, every
piece constant, every body class including its `Execute` path, a `#line` directive for
every mapped statement, and the template's manifest entry. Exactly three things are
elided, uniformly, because they are byte-for-byte identical in every generated file: the
`// <auto-generated/>` header line, the `#pragma warning disable` line after it, and
`global::` qualification on type references (types are shown unqualified). Trailing
`#line hidden`/`#line default` bookkeeping between members is shown once per body rather
than after every brace. The u8 piece twins appear with their gating comment — they exist
only under `HeddleEmitUtf8Pieces=true` + LangVersion ≥ 11 (D15) and are absent from
default v1 output. The snapshot goldens (D19) carry the byte-exact full form of all
seven examples; the manifest entries' `ContentHash` values are the real SHA-256 of each
template source exactly as printed (UTF-8, LF line endings, one trailing newline — the
convention [identity-and-metadata.md](identity-and-metadata.md#the-metadata-format)
states for all worked hashes).

## Example 1 — static text and typed member paths

`views/product.heddle` (options: `Text` profile, `Native` mode, no trim):

```heddle
@model(){{Shop.Product}}@\
<article>
  <h1>@(Name)</h1>
  <p>Made by @(Manufacturer.Name)</p>
</article>
```

Generated `Views_Product.g.cs`, complete:

```csharp

{
    public static class Views_Product
    {
        /// <summary>Typed entry point — the recommended host API (spec D11).</summary>
        public static string Generate(Shop.Product model, object chained = null, object callerData = null)
            => PrecompiledRuntime.GenerateString(Root, model, chained, callerData);

        internal static readonly IProcessStrategy Root = new Body0();

        // Piece table — document order, emitted by the single EmitPiece hook (D15).
        internal const string P0 = "<article>\n  <h1>";
        internal const string P1 = "</h1>\n  <p>Made by ";
        internal const string P2 = "</p>\n</article>\n";
```

Manifest entry (this template is also the first entry of the
[worked manifest](identity-and-metadata.md#the-metadata-format)): key
`"views/product.heddle"`, entry point `typeof(Views_Product)`, model
`typeof(Shop.Product)`, `IsDynamic = false`, `ContentHash`
`"e7bb5b9c393419da6b73abb37a6a7ac1c1e6254462277141fabb27036237b266"`, no imports,
fingerprint `(Text, Native, false)`, bindings `{ "" → Heddle.Extensions.EmptyExtension,
Heddle }`, capabilities `StringOutput`, strategy `Views_Product.Root`.

What to notice: zero parse, zero expression trees — the member accessors the runtime
backend compiles at startup are ordinary C# the *project's* compiler bound (milestone 1);
the `@model` directive and `@\` trim contributed no pieces, by two different mechanisms
(README D22): the `@\` splice and its swallowed characters are excised by the shared
front end (hidden-channel tokens never reach the clean document), while the `@model`
block is a zero-output chain the **emitter's** removal step cuts — the reimplementation
of the back-end `HeddleCompiler.RemoveEmptyItem`, since the runtime backend removes it
in its compile loop, not at parse; `Name` and
`Manufacturer.Name` are reference-typed hops, so `?.` is the exact member-tier
equivalent (protocol rule 6 — a value-typed terminal hop would need the conditional
form, shown in example 2); encoding rides the bound extension, so flipping
`HeddleOutputProfile=Html` changes only the extension type and `RenderType.Encode`
argument, never inlines encoding logic.

## Example 2 — native expressions (phase 1 AST → C#, 1:1)

`views/cart.heddle` (options: `Text`, `Native`; member types on `Shop.Cart`:
`Count : int`, `IsArchived : bool`, `Price : decimal`, `Quantity : int`):

```heddle
@model(){{Shop.Cart}}@\
@if(Count > 0 && !IsArchived){{You have @(Count) items — total @(Price * Quantity).}}
```

The phase 1 `ExprNode` AST is a strict C# subset, so the mapping is mechanical. The full
node-kind table (the phase 1 hierarchy is eight node types — enumerated completely,
against the [phase 1 API contract](../phase-1-native-expressions/README.md#public-api-contract)
and [operator-semantics](../phase-1-native-expressions/operator-semantics.md)):

| `ExprNode` kind | Emitted C# | Notes |
| --- | --- | --- |
| `LiteralNode` | The literal, round-tripped invariant | The one recorded deviation (`-2147483648` typing as `long`) is in phase 1's deviations table and holds for both backends — the emitter reproduces the AST's `Negate(long)` shape. |

**The null-safety rule (normative).** Null-safety is reproduced **hop by hop**, exactly
as `ModelParameter.GetPropertyChainAccessor` emits it — never as a whole-expression

the body; a whole-expression guard would evaluate `false` and skip it — a divergence the
differential harness would catch on the first null-model fixture. Truthiness for `@if`

not compiled into a C# `if`.

`views/card.heddle` — a props definition, an all-constant call site with caller


The generated backend keeps the `object[]` props **carriage** (it is the `Scope`
contract shared with the extension funnel) but replaces the runtime's `PropsBinder`

Rules:

- The definition's prop layout (indices, declared types, defaults) is generation-time
  data; defaults are literals (phase 5's ratified rule), so the prototype array is a
  compile-time constant table.
- An **all-constant call site** shares one `static readonly object[]` — zero per-render
  allocation, phase 5's normative invariant, preserved bit-for-bit.
- A call site with **dynamic arguments** passes `dynamicSetters` — an array of
  `PrecompiledPropSetter` (layout index + `PrecompiledPropEvaluator`, see the support
  API) the engine invokes against `scope.Parent()` after cloning the prototype
  (phase 5 D8's caller-view rule, enforced in one place: the engine, not the emitter).
- **Body prop reads** emit indexed accessors through the engine (the `PropsData` field is
  internal): `PrecompiledRuntime.Prop(in scope, 0)` with the generation-time index —
  same index the layout gave the runtime backend. Prop-first resolution and the `this.`
  escape (phase 5 D9) are **reimplemented in the emitter's resolver** (README D22 —
  the runtime's resolution lives in the back-end member resolver, not the shared front
  end): precedence is syntactic (the layout is declared in the template), so both
  backends resolve identically by rule, differential-gated. The `HED5011` shadowing
  *warning* is milestone-2-only — it needs the model symbol — with the differential
  gate as the interim guarantee that output cannot diverge meanwhile.
- **Caller content is typed by the invoked definition's `:: T`** (body model-typing
  rule): it executes with `ModelData` = the invocation parameter, which is declared to

## Example 6 — a phase 3 branch set

`views/badge.heddle`:

```heddle
@model(){{Shop.Product}}@\
@if(IsFeatured){{ <b>Featured</b> }}
@elif(IsArchived){{ <i>Archived</i> }}
@else(){{ Regular }}
```

Branch semantics are **not** compiled into a C# `if`/`else if` chain — that would inline
extension logic (rejected outright,
[D9](README.md#d9--extension-binding-from-assemblies-at-build-and-runtime-the-match-seam--ratified))
and fork the publish/read protocol. The generated body binds the same three extension
instances the runtime backend would create. Two phase 3 facts shape the emission,
both verified against the phase 3 spec:

- **Whitespace-only text between the blocks of one set is stripped** (phase 3 D10's
  strip machine; non-whitespace gaps additionally warn `HED3001`, positioned at the
  following block's first item). The machine is **back-end** — the private
  `HeddleCompiler.ProcessBranchSets`, which the generator never runs — so the emitter
  **reimplements** it to the same pinned rules (README D22): the newlines between
  `@if`/`@elif`/`@else` are stripped by the emitter's own strip pass and never reach
  the piece table — only the newline *after* the closing `@else` block survives
  (stripping is set-internal only). Each backend applies the identical D10 rules to
  the identical parsed document; the branch corpus differential is the agreement gate.
- **The frame rule:** `ElifExtension`/`ElseExtension` carry `[ScopeChannel]`, so the
  body *hosting* them needs a `ScopeLocals` frame per execution. Here that body is the
  document root, which `Bind` never sees — so the `Root` initializer wraps it in the
  engine's decorator via `PrecompiledRuntime.WithLocalsFrame` (support API). A
  non-root body hosting a set is instead installed with `needsLocals: true` wherever
  *it* is bound. The branch bodies themselves host no participants → `needsLocals:
  false` on all three binds; frame allocation stays zero for templates without
  participants (phase 3 D15's budget).

## Emitter completion notes

- **Definitions, imports, overrides** resolve at generation time in document order,
  exactly as `HeddleCompiler` layers them — the front end contributes the parsed
  definition blocks; the layering/override resolution itself is back-end behavior
  `Emit/DefinitionResolver` reimplements (README D22; the composition fixture —
  `layout.heddle` + `home.heddle` with `<body:body>` — is the differential gate).
  `@<<` needs no runtime linkage: definitions were flattened at generation time; only
  the import hashes reach the manifest (example 3 shows both halves).
- **`@partial`** emits a lazily-initialized strategy field:
  `LazyInitializer.EnsureInitialized(ref _partial0, () =>
  PrecompiledRuntime.ResolvePartial("partials/row.heddle", options))` — registry first,
  dynamic compile second
  ([D7](README.md#d7--runtime-hookup-repeatable-registration-gauntlet-order-lazy-partial-routing)),
  so both mixed-mode directions work.
- **Recursion** (`dynamic-recursion.heddle` corpus): generated strategies may reference
  each other and themselves through the static fields — the CLR type initializer handles
  the cycle because bodies dereference fields at render time, not construction time.
  The recursion-guard semantics stay inside the shared carrier code
  (`DefinitionBaseExtension`'s `ThreadLocal` counter and `>=` guard), but the **limit**
  does not ride `TemplateOptions` — `BindDefinition` bypasses the `InitStart` that
  reads it, so every generated definition call site bakes the build's
  `HeddleMaxRecursionCount` (default 100 — the `TemplateOptions` ctor default) as the
  `maxRecursionCount` argument, and `BindDefinition` assigns it to the carrier
  (README D23; examples 4 and 5 show the baked literal). Build wins over runtime
  options — the same first-compile-wins semantics the dynamic path's options-blind
  resolver cache already has. The differential fixture renders identically on both
  backends and a depth test asserts `TemplateProcessingException` at exactly the baked
  limit.
- **What the emitter never does:** call `Activator`, touch `TemplateFactory` at render
  time, inline extension bodies, emit encoding logic, consult a runtime function
  registry or emit a function call anywhere but the D21 binding targets — the
  `PrecompiledFunctions` shim for defaults, a discovered `[ExportFunctions]`
  container's public static method for exports, or the generated forwarder group for a
  merged overload set — or read files (imports come from `AdditionalFiles` through the
  `ImportReader`, D4; export discovery reads symbol metadata, never assemblies from
  disk).
