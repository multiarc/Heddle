Heddle (Heddle) — language working notes
=========================================

Terse working notes on the *current* language. The full, worked reference lives under docs/
(start at docs/README.md, then docs/language-reference.md). If anything here disagrees with the
ANTLR grammar (src/Heddle.Language/HeddleLexer.g4 + HeddleParser.g4), the grammar wins.


Delimiters
----------
  @% ... %@      definition block
  < name >       definition name          < child : base >   inheritance
  {{ ... }}      body / subtemplate
  -> chain       default output (the definition renders automatically; chain selects its data)
  :: Type        model type annotation (omit it = abstract; see "Late type binding")
  @( ... )       output expression: a member path, or an @-prefixed C# expression
  @name( ... )   extension / definition call            a : b : c   chains calls
  @<<{{ path }}  import another file's definitions
  @{ ... }@      raw block            @: ...           raw to end of line
  @* ... *@      comment              @\  trim following whitespace      @@  literal @


Definition syntax
-----------------
  @%
    < name [: base] >  [ -> chain ]  {{ body }}  [ :: Type ]
    ...more definitions...
  %@

  - Invoke like any extension: @name().  An empty parameter passes the whole current context.
  - A "->" definition is itself an output: it renders in place, at its declaration, using
    `chain` to pick its data. Do NOT also call it by name — that renders it twice.
  - Definitions may be declared at the top level or nested inside a body.


Output syntax
-------------
  @[name]( [parameter] ) [ : name(...) : name(...) ... ] [ {{ body }} ]

  - parameter is one of: a member path, an @-prefixed C# expression, a nested chain, or empty.
  - empty parameter = pass the whole current context value.
  - a chain a:b:c is evaluated RIGHT TO LEFT: c runs first, its output is the "chained" value
    of b, b's output is a's chained, and the LEFTMOST call (a) renders the result. @out() emits
    the chained value, so the left call wraps the output of the call to its right.


Context (how data flows)
------------------------
  - Member paths are relative to the CURRENT model, which narrows as you descend into calls
    (each call rebases the context to its parameter). Like Go text/template (./$) or Mustache.
  - :: (or @root in C#) reaches the original root model.
  - Value calls step back ONE level: the bodies of if/ifnot/date/time/int/money/guid/string
    and for render in the caller's context (scope.Parent()), not the parameter's — that's why
    @if(flag){{ @(Title) }} still sees the surrounding model. No operator climbs further; for
    anything higher, use :: (root) or pass values down explicitly.
  - A second "chained" channel carries the loop index, the chain value, and @out() content.


Late type binding (abstract definitions)
----------------------------------------
  - A definition with NO  :: Type  is abstract (modeled as object). Its type binds late:
      * on use      — the body is compiled against the concrete model at the call site;
      * on narrow   — an inheriting definition pins it with  :: Type.
  - Every use is independently, STATICALLY type-checked. This is the C++ template model
    (re-substitute the concrete model per call site + check it there, no constraints), NOT C#
    generics (compiled once behind constraints, abstracted away, not late-bound). It is also
    not :: dynamic, which defers member resolution to the runtime binder.
  - Type rule — see HeddleCompiler.WalkValidateDefinitionType:
      a child type must be assignable to its base type.
      * abstract (object) base  -> may be narrowed to ANY concrete type;
      * typed base              -> children may only narrow to assignable types;
      * a type fixed anywhere in the chain can't be changed to an incompatible one.
    (e.g. a list field of IEnumerable<Item> has element type Item.)


Examples
--------

Abstract input field — typeless, made concrete wherever it is used:

  @%
    <text>
    {{
      <input type="text" name="@(PropertyName)" value="@(Value)" @out() />
    }}
  %@

Inheritance — fully override `text`, splice in an `attributes` section:

  @%
    @* override the text definition *@
    <text:text>
    {{
      <input type="text" name="@(PropertyName)" id="id_@(PropertyName)"
             value="@(Value)" @attributes(Attributes) />
    }}

    <attributes>
    {{
      @list(){{ @(Name)="@(Value)" }}
    }} :: IEnumerable<NameValue>
  %@

Layout via named regions + @out():

  @%
    <head>   {{ <title>@(Title)</title> }}
    <header> {{ ...site header... }}
    <footer> {{ ...site footer... }}
    <base>
    {{
      <html>
        <head>@head()</head>
        <body>
          @header()
          <main>@out()</main>
          @footer()
        </body>
      </html>
    }} :: PageModel
  %@

  @base(){{ ...page content; lands at @out()... }}

Composition across files (base layout lives in base.heddle):

  @* page.heddle *@
  @<<{{ base.heddle }}            @* import the base layout + its regions *@
  @%
    <header:header>            @* override only what this page changes *@
    {{ ...custom header... }}
  %@
  @base(){{ ...this page's content... }}

Partial — render another template by name, against a member:

  @partial(Property){{ partialName }}
